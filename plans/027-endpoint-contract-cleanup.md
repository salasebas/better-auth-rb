## Why this matters

`reference/endpoints-inventory.md` currently exposes a few avoidable contract
problems:

- one deprecated password-reset alias that should be removed instead of carried
  forward;
- duplicated body parameter names where both camelCase and snake_case are
  advertised as first-class public JSON fields;
- Stripe subscription endpoints whose security-sensitive redirect/query fields
  are used by code but invisible in the generated endpoint inventory;
- passkey challenge-generation endpoints that use GET while writing cookies and
  verification records.

Do **not** turn this into a broad REST redesign. The Ruby port should still
match Better Auth's action-style API where that is upstream behavior. The goal
is to reduce ambiguous public contracts and remove unsafe or deprecated edges
without renaming standards-owned protocol surfaces.

## Proposed endpoint contract changes

Only implement the following changes. Do not rename routes that are not listed
here.

| Area | Current | New contract | Why |
| --- | --- | --- | --- |
| Email OTP password reset | `POST /forget-password/email-otp` and `POST /email-otp/request-password-reset` | Remove `POST /forget-password/email-otp`; keep `POST /email-otp/request-password-reset` with body `{ "email": string }` | The canonical route already exists, upstream marks the old route deprecated, and keeping both doubles public attack/documentation surface. |
| Public JSON field names | Many action endpoints advertise both camelCase and snake_case body fields | Advertise one canonical wire field per semantic input. Prefer upstream camelCase for Better Auth action endpoints; keep RFC/protocol names exactly where specs require them. Runtime may continue accepting Ruby snake_case aliases unless explicitly removed. | Generated docs/OpenAPI should not make clients guess which duplicate field wins. Ruby convenience aliases are fine as compatibility behavior, not as primary public contract. |
| Stripe subscription routes | Inventory shows no inputs for subscription routes | Add complete OpenAPI request/query metadata for Stripe subscription endpoints, including redirect URL fields and customer/reference/subscription fields | These are public, security-sensitive inputs. The inventory should show what is validated and what clients must send. |
| Passkey challenge generation | `GET /passkey/generate-register-options` and `GET /passkey/generate-authenticate-options` write cookies and verification records | Prefer `POST` for both challenge-generation endpoints. Move registration inputs from query to JSON body. Authentication can use an empty JSON body. Remove GET, or keep it for one release only as hidden/deprecated with `Cache-Control: no-store`. | GET should be safe and cache/prefetch tolerant. These endpoints mutate server state and create authentication challenges. |

## Non-changes

These were considered and rejected for this plan:

- No broad REST rewrite of action endpoints such as `/admin/get-user`,
  `/organization/update`, `/api-key/create`, or `/two-factor/verify-otp`.
  Better Auth upstream exposes action-style routes, and changing all of them
  would break parity with little security benefit.
- No rename of SCIM `/scim/v2/*` routes or SCIM parameter casing.
- No rename of OAuth/OIDC/device protocol fields such as `client_id`,
  `redirect_uri`, `grant_type`, `device_code`, or `user_code`.
- No rename of SAML protocol fields `SAMLRequest`, `SAMLResponse`, and
  `RelayState`.
- No duplicate implementation of `/mcp/*` and OAuth Provider legacy alias
  removal. That belongs to Plan 019, with the deprecation matrix in Plan 026.

### Step 4: Move passkey challenge generation from GET to POST

This is the only recommended change that intentionally diverges from upstream
v1.6.9. Execute it only if maintainers accept the HTTP-semantics/security trade.

1. In passkey route metadata and handlers:
   - change `/passkey/generate-register-options` to `POST`;
   - change `/passkey/generate-authenticate-options` to `POST`;
   - read registration options from JSON body instead of query;
   - pass the normalized body/query context into `Challenges.store_challenge`
     so `context` is still stored correctly.
2. Decide the migration policy before coding:
   - preferred breaking change: remove GET routes entirely;
   - one-release migration: keep GET as hidden/deprecated, return
     `Cache-Control: no-store`, and document removal date.
3. Update passkey docs/tests:
   - POST generation succeeds and stores the challenge;
   - GET generation is 404/405, or hidden/deprecated with `Cache-Control:
     no-store` if the migration policy keeps it temporarily;
   - `verify-registration` and `verify-authentication` still consume the stored
     challenge.
4. Regenerate inventory.
