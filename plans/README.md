## Findings Considered And Rejected

- Upstream `client/**`: rejected for this plan set because the Ruby project is
  server packages only and has no Better Auth browser client package.
- Upstream `types/types.test.ts`: rejected except for behavior that also exists
  at runtime; TypeScript type inference has no Ruby server equivalent.
- Upstream `integrations/next-js.test.ts`: rejected because framework adapters
  in this repo are Ruby packages and Next.js is not a Ruby server target.
- Upstream `plugins/organization/client.test.ts`: rejected for this plan set
  because it asserts client plugin API shape, not server organization behavior.
- Upstream `plugins/mcp/client/**`: rejected for this plan set because it is
  client-side MCP behavior; server MCP tests can be inventoried separately if
  maintainers decide MCP is part of the current parity target.
- External plugin packages already near or above upstream coverage (`api-key`,
  `passkey`, `scim`, `stripe`, `sso`) are rejected from this core plan set; keep
  their coverage in their package-specific suites.
- Broad refactor of duplicated helpers in existing tests: rejected because it
  adds review risk without increasing upstream parity. Plan 006 adds helpers for
  new tests only.
- Global monkeypatching of `BetterAuth::Password` for speed: rejected because
  it can hide password-hashing regressions. Plan 014 uses explicit test auth
  config callbacks only in tests that do not assert password behavior.
- Moving service-backed tests out of coverage entirely: rejected. Plan 015 moves
  Redis, MongoDB, and SQL adapter coverage to a deliberate integration workflow
  rather than deleting or silently skipping it.
- Broad mocking of Stripe/OIDC/SAML/network behavior: rejected for now. The
  measured suites were not dominated by real network calls, and existing tests
  already use local fakes/stubs where those external systems are involved.
- Parallelizing shared database integration tests inside a single job: rejected
  for the first pass because many tests reset shared schemas. Plan 015 favors
  explicit workflow/job separation before introducing concurrency.
