This page contains information about security features of RubyAuth.

## Password Hashing

RubyAuth uses the `scrypt` algorithm to hash passwords by default. This algorithm is designed to be memory-hard and CPU-intensive, making it resistant to brute-force attacks. You can customize the password hashing function by setting the `password` option in the configuration. This option should include a `hash` function to hash passwords and a `verify` function to verify them.

## Secret Rotation

RubyAuth supports non-destructive rotation of `BETTER_AUTH_SECRET`. When you configure versioned secrets via the `secrets` option or the `BETTER_AUTH_SECRETS` environment variable, all new encrypted data includes a key version identifier. Decryption performs a direct key lookup by version — no trial decryption.

Legacy data encrypted before rotation (bare-hex format) is still decryptable using the original `BETTER_AUTH_SECRET` as a fallback. No database migrations or downtime are required. Data is lazily re-encrypted with the current key when it is next written.

See the [`secrets` option](/docs/reference/options#secrets) for configuration details.

## Session Management

### Session Expiration

RubyAuth uses secure session management to protect user data. Stateful
deployments store sessions in the configured database or secondary storage;
stateless deployments use the signed session cookie as their authority and
therefore cannot revoke that cookie through a server-side session store. By
default, sessions expire after 7 days, but you can customize this value in the
configuration. Additionally, each time a session is used, if it reaches the
`update_age` threshold, the expiration date is extended, which by default is
set to 1 day.

### Session Revocation

RubyAuth allows you to revoke sessions to enhance security. When a session is revoked, the user is logged out and can no longer access the application. A logged in user can also revoke their own sessions to log out from different devices or browsers.

Endpoints that perform sensitive work, including password, account, session-revocation, and administrator operations, use an authoritative session read. When a database or secondary session store is configured, these endpoints bypass both the cookie cache and any session memoized earlier in the request. Stateless deployments continue to treat the signed session cookie as the authority, but refreshing its cache never extends the session beyond its original expiration.

See the [session management](/docs/concepts/session-management) for more details.

## CSRF Protection

RubyAuth includes multiple safeguards to prevent Cross-Site Request Forgery (CSRF) attacks:

1. **Avoid simple requests**
   See [Avoiding simple requests](https://developer.mozilla.org/en-US/docs/Web/Security/Attacks/CSRF#avoiding_simple_requests) for more details. RubyAuth only allows requests with a non-simple header or a `Content-Type` header of `application/json`.

2. **Origin Validation**
   Each request’s `Origin` header is verified to confirm it comes from your application or another explicitly trusted source. Requests from untrusted origins are rejected. By default, RubyAuth trusts the base URL of your app, but you can specify additional trusted origins via the `trusted_origins` configuration option.

3. **Secure Cookie Settings**
   Session cookies use the `SameSite=Lax` attribute by default, preventing browsers from sending cookies with most cross-site requests. You can override this behavior using the `default_cookie_attributes` option.

4. **Fetch Metadata Protection for First-Login CSRF**
   RubyAuth also uses [Fetch Metadata headers](https://developer.mozilla.org/en-US/docs/Web/Security/Attacks/CSRF#fetch_metadata) (`Sec-Fetch-Site`, `Sec-Fetch-Mode`, `Sec-Fetch-Dest`) to provide additional CSRF protection specifically for **first-login scenarios**, where the client does not yet have any cookies.

   This protection applies to **sign-in and sign-up email routes**, which accept form submissions (simple requests) to support progressive enhancement patterns.

   When a sign-in or sign-up request is received **without** any cookies:

   * If the browser indicates a **cross-site navigation** (for example: `Sec-Fetch-Site: cross-site` and `Sec-Fetch-Mode: navigate`), RubyAuth **blocks** the request as a potential login CSRF attack.

   * If the request is **same-origin** or **same-site**, RubyAuth still validates the `Origin`/`Referer` against your `trusted_origins` configuration, using the existing origin validation logic.

   * If the client does **not** send Fetch Metadata headers but does send an `Origin` or `Referer`, RubyAuth still validates that value for sign-in and sign-up requests. Requests with no cookies, Fetch Metadata, `Origin`, or `Referer` retain the non-browser fallback.

   This mechanism allows modern browsers to receive stronger protection against first-login CSRF attacks **without requiring CSRF tokens or client-side JavaScript**, and works seamlessly with progressive enhancement patterns.

   <Callout type="info">
     Non-browser HTTP clients (mobile apps, server-to-server) should set the `Origin` or `Referer` header appropriately when making requests that include cookies. This ensures proper CSRF validation regardless of Fetch Metadata support.
   </Callout>

5. **No Mutations on GET Requests (with additional safeguards)**
   `GET` requests are assumed to be read-only and should not alter the application's state. In cases where a `GET` request must perform a mutation—such as during OAuth callbacks - RubyAuth applies extra security measures, including validating `nonce` and `state` parameters to ensure the request's authenticity.

### Disabling Security Checks

RubyAuth exposes two advanced escape hatches. Both weaken request validation and
should be limited to narrowly understood compatibility cases.

#### `disable_csrf_check`

Disables **all CSRF protection**, including:

* Origin header validation when cookies are present
* Fetch Metadata checks (`Sec-Fetch-Site`, `Sec-Fetch-Mode`, `Sec-Fetch-Dest`)
* Cross-site navigation blocking for first-login scenarios

```ruby
require "better_auth"

auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  base_url: ENV.fetch("BETTER_AUTH_URL", "http://localhost:3000"),
  database: :postgres,
  advanced: { disable_csrf_check: true }
)
```

<Callout type="warning">
  Disabling CSRF checks allows requests from any origin to use cookies and perform actions on behalf of users. This opens your application to CSRF attacks.
</Callout>

#### `disable_origin_check`

Disables **URL validation** against `trusted_origins`, including:

* `callback_url` validation
* `redirectTo` validation
* `error_callback_url` validation
* `new_user_callback_url` validation

```ruby
require "better_auth"

auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  base_url: ENV.fetch("BETTER_AUTH_URL", "http://localhost:3000"),
  database: :postgres,
  advanced: { disable_origin_check: true, disable_csrf_check: false }
)
```

<Callout type="warning">
  Disabling origin checks allows any URL to be used in redirects and callbacks. This opens your application to open redirect vulnerabilities.
</Callout>

<Callout type="info">
  For backward compatibility, `disable_origin_check: true` also disables CSRF
  protection only when `disable_csrf_check` is omitted. Set
  `disable_csrf_check: false` explicitly to keep CSRF validation enabled. An
  array value such as `disable_origin_check: ["/public"]` skips ordinary origin
  and redirect-URL validation for matching path prefixes. Fetch Metadata still
  blocks a first-login cross-site navigation on those paths unless
  `disable_csrf_check: true` disables that protection globally.
</Callout>

#### Summary

| Option               | What it disables                                                       |
| -------------------- | ---------------------------------------------------------------------- |
| `disable_csrf_check` | CSRF origin and Fetch Metadata checks |
| `disable_origin_check` | Callback/redirect URL validation; also CSRF checks only under the deprecated omitted-`disable_csrf_check` compatibility behavior |

## OAuth State and PKCE

OAuth state prevents CSRF and PKCE protects the authorization-code exchange.
By default (`account.store_state_strategy: "cookie"`), RubyAuth stores encrypted
state data and the PKCE verifier in a short-lived cookie and consumes that
cookie during the callback. With the database strategy, RubyAuth stores the
state package as a short-lived verification record and also checks a signed
state cookie when a browser request is available. Successful reads consume the
stored state so it cannot be replayed.

## Cookies

RubyAuth assigns secure cookies by default when the base URL uses `https`. These secure cookies are encrypted and only sent over secure connections, adding an extra layer of protection. They are also set with the `sameSite` attribute to `lax` by default to prevent cross-site request forgery attacks. And the `httpOnly` attribute is enabled to prevent client-side JavaScript from accessing the cookie.

During cookie migrations, a client may temporarily send both a legacy cookie name and its `__Secure-` equivalent. RubyAuth gives the `__Secure-` cookie precedence. When a session or account cookie is expired, earlier same-name `Set-Cookie` values and chunked variants are removed from the response before the expiration value is emitted.

For Cross-Subdomain Cookies, you can set the `cross_sub_domain_cookies` option in the configuration. This option allows cookies to be shared across subdomains, enabling seamless authentication across multiple subdomains.

### Customizing Cookies

You can customize cookie names to minimize the risk of fingerprinting attacks and set specific cookie options as needed for additional control. For more information, refer to the [cookie options](/docs/concepts/cookies).

Plugins can also set custom cookie options to align with specific security needs. If you're using RubyAuth in non-browser environments, plugins offer ways to manage cookies securely in those contexts as well.

## Rate Limiting

RubyAuth includes built-in rate limiting to safeguard against brute-force attacks. Rate limits are applied across all routes by default, with specific routes subject to stricter limits based on potential risk.

## IP Address Headers

Better Auth Ruby uses client IP addresses for rate limiting and session metadata. By default, it relies on the address resolved by the hosting framework or Rack and does not directly trust raw forwarded headers.

If a reverse proxy overwrites a dedicated single-value client-IP header, opt in to that trusted header explicitly:

```ruby
require "better_auth"

auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  base_url: ENV.fetch("BETTER_AUTH_URL", "http://localhost:3000"),
  database: :postgres,
  trusted_origins: ["https://*.example.com"],
  advanced: {
    ip_address: {
      ip_address_headers: ["x-real-ip"]
    }
  },
  rate_limit: {enabled: true, window: 10, max: 100}
)
```

For a comma-separated header such as `X-Forwarded-For`, configure both the header and the actual proxy IP addresses or CIDR ranges:

```ruby
advanced: {
  ip_address: {
    ip_address_headers: ["x-forwarded-for"],
    trusted_proxies: ["192.0.2.10", "10.0.0.0/24"]
  }
}
```

Better Auth first verifies that the direct network peer matches `trusted_proxies`. It then walks the chain from right to left, skips trusted hops, and selects the first untrusted IP. A missing or untrusted peer, malformed proxy configuration, malformed chain, chain containing only trusted proxies, or multi-hop chain without valid `trusted_proxies` is rejected. If no safe framework or direct-peer fallback exists, rate limiting uses one shared per-path bucket instead of being skipped.

<Callout type="info">
  **Important**

  * A configured IP address header is an explicit trust decision. Ensure your proxy or load balancer overwrites or sanitizes it and that end users cannot reach the origin directly.
  * `trusted_proxies` verifies the direct peer before interpreting a chain. Use the narrow addresses of your actual proxies rather than broad private ranges that may also include clients.
  * In dev/test environments, if the IP cannot be retrieved from headers, 127.0.0.1 is used as a fallback.
</Callout>

<Callout type="warning">
  Earlier Better Auth Ruby releases trusted the leftmost `X-Forwarded-For` value by default. After upgrading, configure framework-level trusted proxies or an explicit ingress-controlled header. A missing or rejected address now fails closed into a shared bucket, so validate the deployment before enabling production traffic.
</Callout>

## Canonical and request-serving origins

RubyAuth requires a static canonical `base_url` (or `BETTER_AUTH_URL`). Security-sensitive identity values such as OAuth/OIDC issuer metadata, provider callback defaults, JWT issuer/audience, and OpenAPI server metadata always use that canonical URL and never use request headers.

For approved multi-domain deployments, list full origins in `serving_origins`. RubyAuth may use a matching request origin for self-referential capability links such as verification, password reset, deletion, and magic links. An unknown or malformed host falls back to the canonical URL and is never added to trusted origins.

```ruby
auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  base_url: "https://auth.example.com",
  serving_origins: ["https://tenant.example.com", "https://*.preview.example.com"],
  trusted_origins: ["https://frontend.example.com"],
  advanced: { trusted_proxy_headers: true }
)
```

`serving_origins` are automatically trusted for CSRF and redirect validation. A `trusted_origins` entry is not promoted to serving status, so a frontend-only origin cannot control token-bearing URLs.

### Trusted proxy headers

By default, `X-Forwarded-Host` and `X-Forwarded-Proto` are ignored. Set `advanced.trusted_proxy_headers: true` only when a trusted edge proxy overwrites both headers and prevents clients from supplying them directly. Even then, the resulting full origin must match `serving_origins`; syntactic header validation alone is not authorization.

Request-specific state is isolated and cleared after every Rack or direct `auth.api` call. One request can never change the canonical URL or select a serving origin for a later request.

See [Multiple Serving Origins](/docs/guides/dynamic-base-url) for deployment and migration guidance.

## Trusted Origins

Trusted origins prevent CSRF attacks and block open redirects. You can set a list of trusted origins in the `trusted_origins` configuration option. Requests from origins not on this list are automatically blocked.

### Basic Usage

The most basic usage is to specify exact origins, below is an example of a trusted origins configuration:

```ruby
require "better_auth"

auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  base_url: ENV.fetch("BETTER_AUTH_URL", "http://localhost:3000"),
  database: :postgres,
  trusted_origins: ["https://app.example.com"],
  rate_limit: { enabled: true, window: 10, max: 100 }
)
```

<Callout type="warning">
  Do not leave the localhost origin in a trusted origins list of a production auth instance.
</Callout>

### Wildcard Origins

RubyAuth supports wildcard patterns in trusted origins, which allows you to trust multiple subdomains with a single entry:

```ruby
require "better_auth"

auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  base_url: ENV.fetch("BETTER_AUTH_URL", "http://localhost:3000"),
  database: :postgres,
  trusted_origins: ["https://*.example.com"],
  rate_limit: { enabled: true, window: 10, max: 100 }
)
```

#### Protocol-specific wildcards

When using a wildcard pattern with a protocol prefix (like `https://`):

* The protocol must match exactly
* The domain can have any subdomain in place of the `*`
* Requests using a different protocol will be rejected, even if the domain matches

#### Protocol-agnostic wildcards

When using a wildcard pattern without a protocol prefix (like `*.example.com`):

* Any protocol (http, https, etc.) will be accepted
* The domain must match the wildcard pattern

### Custom Schemes

Trusted origins also support custom schemes for mobile apps and browser extensions:

```ruby
require "better_auth"

auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  base_url: ENV.fetch("BETTER_AUTH_URL", "http://localhost:3000"),
  database: :postgres,
  trusted_origins: ["myapp://auth-callback"],
  rate_limit: { enabled: true, window: 10, max: 100 }
)
```

### Dynamic origin list

You can also dynamically set the list of trusted origins by providing a function that returns it:

```ruby
require "better_auth"

trusted_for_request = lambda do |request|
  request.get_header("HTTP_X_TENANT") == "acme" ? ["https://acme.example.com"] : []
end

auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  base_url: ENV.fetch("BETTER_AUTH_URL", "http://localhost:3000"),
  database: :postgres,
  trusted_origins: trusted_for_request,
  rate_limit: { enabled: true, window: 10, max: 100 }
)
```

<Callout>
  **Important**: This function will be invoked per incoming request, so be careful if you decide to dynamically fetch your list of trusted domains.
</Callout>

## Email Enumeration Protection

RubyAuth prevents email enumeration on the sign-up and change-email endpoints. When `require_email_verification` is enabled or `auto_sign_in` is set to `false`, the sign-up endpoint returns the same `200` response whether the email is already registered or not, following [OWASP authentication best practices](https://cheatsheetseries.owasp.org/cheatsheets/Authentication_Cheat_Sheet.html#authentication-and-error-messages). Timing attacks are mitigated by simulating password hashing on duplicate sign-up attempts.

See [Email and Password — Email Enumeration Protection](/docs/authentication/email-password#email-enumeration-protection) for configuration details.
