## How do I look up the current session on the server?

Use the server API or the framework helper for your integration. In plain Rack,
pass the request headers to `auth.api.get_session`; in Rails, prefer the
`better_auth-rails` helper methods documented by the Rails integration.

```ruby
session = auth.api.get_session(headers: request.env)
user = session&.dig(:user)
```

Do not trust client-provided user IDs. Read the session from the RubyAuth cookie
or bearer token on every request that needs authentication.

## Why do OAuth callbacks fail in production but work locally?

Most callback failures come from a mismatch between the URL RubyAuth uses and
the URL registered with the provider. Check `base_url`, `base_path`,
`trusted_origins`, proxy headers, and the callback URL configured in the OAuth
or SSO dashboard.

```ruby
BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  base_url: "https://app.example.com",
  base_path: "/api/auth",
  trusted_origins: ["https://app.example.com"],
  database: :postgres
)
```

When the app runs behind a proxy, make sure the proxy preserves `Host`,
`X-Forwarded-Host`, and `X-Forwarded-Proto` consistently.

## Can I use preview deployments or multiple domains?

Yes. Use the dynamic `base_url` object and keep the allowlist narrow. See the
[dynamic base URL guide](/docs/guides/dynamic-base-url) for deployment patterns.

```ruby
base_url: {
  allowed_hosts: ["app.example.com", "*.preview.example.com"],
  protocol: "https",
  fallback: "https://app.example.com"
}
```

## How do I diagnose duplicate gem or version issues?

Use Bundler. The docs site uses Node tooling internally, but runtime RubyAuth
apps should diagnose Ruby gem versions with Bundler.

```bash
bundle info better_auth
bundle exec ruby -e 'require "better_auth"; puts BetterAuth::VERSION'
bundle exec ruby -e 'puts Gem.loaded_specs.select { |name, _| name.start_with?("better_auth") }.keys'
```

If multiple local checkouts or path gems are involved, inspect `Gemfile.lock`
and any `path:` entries in the application Gemfile.

## Which test framework should I use?

Core RubyAuth uses Minitest. Adapter and plugin packages in this repository use
RSpec where that package already chose RSpec. Match the package you are editing,
and run the workspace checks before submitting changes.

```bash
bundle exec rake test
bundle exec standardrb
cd docs-site && pnpm lint && pnpm build
```

## How should Rails apps mount RubyAuth?

Use `better_auth-rails` when you are in Rails. Configure the initializer and
mount the generated route at the same path as `config.base_path`.

```ruby title="config/initializers/better_auth.rb"
BetterAuth::Rails.configure do |config|
  config.secret = ENV.fetch("BETTER_AUTH_SECRET")
  config.base_url = ENV.fetch("BETTER_AUTH_URL")
  config.base_path = "/api/auth"
  config.database = :postgres
end
```

## How should plain Rack apps mount RubyAuth?

Build one `auth` instance at boot and route requests to it. Avoid rebuilding the
auth object per request because plugins, adapters, and telemetry are intended to
be initialized once.

```ruby title="config.ru"
require "better_auth"

auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  base_url: ENV.fetch("BETTER_AUTH_URL"),
  database: :postgres
)

map "/api/auth" do
  run auth
end
```

## How do I reset local development data?

For database-backed development, reset the application database the same way you
reset the rest of your app. Then rerun the RubyAuth migrations or schema
generator for your adapter.

```bash
bundle exec better-auth migrate --cwd . --config config/better_auth.rb --yes
```

For stateless or memory-backed experiments, restart the process and clear the
browser cookies for the local auth host.

## Why do I get CSRF or invalid origin errors?

RubyAuth validates origins on browser-facing routes. Add every trusted frontend
origin explicitly, including local development ports.

```ruby
trusted_origins: [
  "http://localhost:3000",
  "http://localhost:5173",
  "https://app.example.com"
]
```

Do not use broad wildcard origins for production apps unless you also control
the matching domains.

## Where should secrets live?

Use `BETTER_AUTH_SECRET` or your framework's credential manager. The secret must
be stable across all running app instances and should be at least 32 characters
with high entropy.

```bash
bundle exec better-auth secret
```

Rotating secrets during an OAuth or SSO flow can invalidate state cookies and
produce callback errors. Use the `secrets` rotation option when you need a
graceful rollout.
