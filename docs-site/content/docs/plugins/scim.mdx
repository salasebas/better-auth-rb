title: System for Cross-domain Identity Management (SCIM)

<RubyAuthDisclaimer />

The `better_auth-scim` gem exposes a SCIM 2.0 user-provisioning API for identity
providers. SCIM provisions accounts; it is not a login protocol and does not
require the SSO plugin.

## Installation

Add the gem and mount the same RubyAuth Rack handler used by the rest of your
application:

```ruby title="Gemfile"
gem "better_auth-scim"
```

```ruby title="config/auth.rb"
require "better_auth"
require "better_auth/scim"

auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  base_url: ENV.fetch("BETTER_AUTH_URL"),
  database: :postgres,
  plugins: [BetterAuth::Plugins.scim]
)
```

Run the normal RubyAuth migration or schema-generation command after adding the
plugin. The SCIM routes require `GET`, `POST`, `PUT`, `PATCH`, and `DELETE`, all
of which work through the Rack handler.

## Supported surface

The SCIM base URL is `${BETTER_AUTH_URL}/api/auth/scim/v2` when the default
RubyAuth base path is used. The plugin exposes:

| Method | Path | Purpose |
| --- | --- | --- |
| `GET` | `/scim/v2/Users` | List users within the token boundary |
| `GET` | `/scim/v2/Users/:userId` | Read one user |
| `POST` | `/scim/v2/Users` | Provision a user |
| `PUT` | `/scim/v2/Users/:userId` | Replace supported user attributes |
| `PATCH` | `/scim/v2/Users/:userId` | Apply SCIM PatchOp operations |
| `DELETE` | `/scim/v2/Users/:userId` | Deprovision a user |
| `GET` | `/scim/v2/ServiceProviderConfig` | Read server capabilities |
| `GET` | `/scim/v2/Schemas` | List supported schemas |
| `GET` | `/scim/v2/Schemas/:schemaId` | Read one schema |
| `GET` | `/scim/v2/ResourceTypes` | List resource types |
| `GET` | `/scim/v2/ResourceTypes/:resourceTypeId` | Read one resource type |

<Callout type="info">
  **Group support erratum:** an earlier package changelog incorrectly claimed
  SCIM Group provisioning parity. Better Auth v1.6.23 exposes no SCIM Group
  routes, and neither does this Ruby plugin. Only User and metadata resources
  are supported.
</Callout>

## Provider tokens

Create a provider connection with an authenticated RubyAuth session:

```http
POST /api/auth/scim/generate-token
Content-Type: application/json

{
  "providerId": "acme-directory",
  "organizationId": "org_123"
}
```

`organizationId` is optional. When supplied, the organization plugin must be
enabled and the caller must be an organization member with an allowed role.
The default allowed roles are `admin` and the configured organization creator
role (`owner` by default). Provider identifiers cannot collide with built-in,
configured social, resolved, or SSO provider identifiers.

Generated provider tokens are stored as hashes by default. The clear bearer
token is returned only when it is created or rotated, so store it in the
identity provider at that time. The bearer credential sent to SCIM endpoints is
the URL-safe Base64 encoding of:

```text
token:provider_id[:organization_id]
```

Do not log either the clear token or its encoded bearer value.

### Static tokens

`default_scim` can define static connections without database records. Static
secrets live in application configuration and are not protected by the generated
token hashing default. Prefer generated tokens in production. If a static token
is necessary, load it from a secret manager or environment variable, scope it
to a dedicated provider (and organization when applicable), rotate it
deliberately, and never commit it:

```ruby title="config/auth.rb"
BetterAuth::Plugins.scim(
  default_scim: [
    {
      provider_id: "bootstrap-directory",
      scim_token: ENV.fetch("SCIM_BOOTSTRAP_TOKEN"),
      organization_id: ENV.fetch("SCIM_BOOTSTRAP_ORGANIZATION_ID")
    }
  ]
)
```

## Authorization and identity safety

Every SCIM bearer token is limited to its provider and optional organization.
List, read, update, and delete operations enforce that boundary.

Existing-email linking is disabled by default: provisioning an email already
owned by another user returns SCIM `409`. Enable linking only with explicit
positive constraints:

```ruby title="config/auth.rb"
BetterAuth::Plugins.scim(
  link_existing_users: {
    trusted_domains: ["example.com"],
    require_existing_org_membership: true
  },
  can_generate_token: ->(payload) { !payload[:member].nil? }
)
```

Every supplied linking constraint must pass, and an empty policy hash denies
linking. `link_existing_users: true` restores unrestricted legacy linking and
should be used only when every issuer is trusted to take control of existing
matching identities. `can_generate_token` runs after built-in ownership,
membership, and role checks and can only further restrict issuance.

Personal provider connections are owned by their creator by default. Set
`provider_ownership: { enabled: false }` only for a deliberate legacy shared
management model in which every authenticated manager is trusted.

## Update and deactivation behavior

Both `PUT` and `PATCH` normalize changed email addresses to lowercase and reject
an address already used by another user. A real email change clears email
verification; changing only letter case preserves it.

Setting `active: false` through `PUT` or `PATCH` requires the admin plugin. It
uses the admin banned state and revokes all sessions. Without the admin plugin,
deactivation fails instead of silently leaving an active account.

For organization-scoped tokens, `DELETE` removes the organization membership,
team memberships in that organization, and the current SCIM account while
preserving the global user. For personal tokens, it unlinks the current SCIM
account and deletes the global user only when no other identity remains.
Deleting a provider connection invalidates that token but does not delete users
that were already provisioned.

## Configuration

All plugin option names use Ruby snake_case:

| Option | Behavior |
| --- | --- |
| `store_scim_token` | Generated-token storage; defaults to `"hashed"`. Also accepts `"encrypted"`, `"plain"`, or a custom hash/encrypt strategy. |
| `default_scim` | Static provider entries using `provider_id`, `scim_token`, and optional `organization_id`. |
| `provider_ownership` | Personal connection ownership; defaults to `{ enabled: true }`. |
| `required_role` | Allowed organization roles; defaults to `admin` plus the creator role. |
| `link_existing_users` | Existing-email linking policy; disabled by default. |
| `can_generate_token` | Additional issuance callback receiving user, provider, organization, and membership context. |
| `before_scim_token_generated` | Hook called before generated-token persistence. |
| `after_scim_token_generated` | Hook called after generated-token persistence. |

A custom one-way token hasher can be supplied as follows:

```ruby title="config/auth.rb"
BetterAuth::Plugins.scim(
  store_scim_token: {
    hash: ->(token) { MyTokenHasher.hash(token) }
  }
)
```

If you use reversible custom encryption, provide both `encrypt` and `decrypt`
callbacks. Prefer the default hashed mode unless the clear provider token must
be recovered for a specific integration.

## Schema

The plugin adds a `scimProvider` model (mapped to `scim_providers` by the Ruby
schema layer) with provider ID, stored token value, optional organization ID,
and—when provider ownership is enabled—optional owner user ID. Applications
should also enforce a unique composite index on account provider ID and account
ID using their database adapter's migration facilities.
