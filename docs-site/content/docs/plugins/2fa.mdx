`OTP` `TOTP` `Backup Codes` `Trusted Devices`

Two-Factor Authentication (2FA) adds an extra security step when users log in. Instead of just using a password, they'll need to provide a second form of verification. This makes it much harder for unauthorized people to access accounts, even if they've somehow gotten the password.

This plugin offers two main methods to do a second factor verification:

1. **OTP (One-Time Password)**: A temporary code sent to the user's email or phone.
2. **TOTP (Time-based One-Time Password)**: A code generated by an app on the user's device.

**Additional features include:**

* Generating backup codes for account recovery
* Enabling/disabling 2FA
* Managing trusted devices

## Installation

<Steps>
  <Step>
    ### Add the plugin to your auth config

    Add the two-factor plugin to your auth configuration and specify your app name as the issuer.

    ```ruby title="config/auth.rb"
    require "better_auth"

    auth = BetterAuth.auth(
      secret: ENV.fetch("BETTER_AUTH_SECRET"),
      base_url: ENV.fetch("BETTER_AUTH_URL", "http://localhost:3000"),
      plugins: [
        BetterAuth::Plugins.two_factor(issuer: "My App")
      ]
    )
    ```
  </Step>

  <Step>
    ### Migrate the database

    Run the migration or generate the schema to add the necessary fields and tables to the database.

## Usage

### Enabling 2FA

To enable two-factor authentication, call `twoFactor.enable` with the user's password (required for credential accounts) and issuer (optional). If you enable `allowPasswordless`, the password can be omitted for users without a credential account.

<APIMethod path="/two-factor/enable" method="POST" requireSession>
  ```
  type enableTwoFactor = {
      /**
       * The user password for credential accounts.
       */
      password?: string = "password123"
      /**
       * Custom issuer for the TOTP URI.
       */
      issuer?: string = "My App"
  }
  ```
</APIMethod>

When 2FA is enabled:

* An encrypted `secret` and `backupCodes` are generated.
* `enable` returns `totpURI` and `backupCodes`.

Note: `twoFactorEnabled` won’t be set to `true` until the user verifies their TOTP code. Learn more about verifying TOTP [here](#totp). You can skip verification by setting `skipVerificationOnEnable` to true in your plugin config.

<Callout type="warn">
  By default, Two Factor requires a credential (password) account. To allow passwordless users (passkeys, magic links, email OTP, OAuth/social, or anonymous), set `allowPasswordless: true`.
</Callout>

### Sign In with 2FA

When a user with 2FA enabled tries to sign in via email, the response object will contain `twoFactorRedirect` set to `true` and `twoFactorMethods` — an array of the 2FA methods available for the user (e.g. `["totp"]`, `["totp", "otp"]`). Use `twoFactorMethods` to decide which verification UI to show.

Handle the second factor challenge on your server by checking the `twoFactorRedirect` response from `auth.api.sign_in_email`, then forwarding the returned two-factor cookie into the relevant verification call.

<Callout type="warn">
  **With `auth.api`**

  When you call `auth.api.sign_in_email` on the server, and the user has 2FA enabled, it will return an object where `twoFactorRedirect` is set to `true`. Check whether `twoFactorRedirect` is true before continuing the flow.


  If you chain multiple `auth.api.*` calls in the same server flow, make sure you forward the cookies from the previous auth response into the next call.

  ```ruby title="server.rb"
  status, headers, body = auth.api.sign_in_email(
    body: {email: "ada@example.com", password: "password123"},
    as_response: true
  )

  if body[:twoFactorRedirect]
    two_factor_cookie = headers.fetch("set-cookie")
    auth.api.verify_totp(
      headers: {"cookie" => two_factor_cookie},
      body: {code: params.fetch("code")}
    )
  end
  ```
</Callout>

During a sign-in challenge, TOTP and backup-code verification share a fixed budget of five failed attempts. Switching between those two methods does not reset the budget. After it is exhausted, the challenge is invalidated and the user must sign in again. OTP codes use their own configurable `allowedAttempts` budget instead (five by default).

These per-challenge budgets are separate from the account-wide lockout described under [Account lockout](#account-lockout). A failed TOTP, OTP, or backup-code verification consumes both its applicable challenge budget and the shared account counter.

### Disabling 2FA

To disable two-factor authentication, call `twoFactor.disable` with the user's password (required for credential accounts). If you enable `allowPasswordless`, the password can be omitted for users without a credential account.

<APIMethod path="/two-factor/disable" method="POST" requireSession>
  ```
  type disableTwoFactor = {
      /**
       * The user password for credential accounts.
       */
      password?: string = "password123"
  }
  ```
</APIMethod>

### TOTP

TOTP (Time-Based One-Time Password) is an algorithm that generates a unique password for each login attempt using time as a counter. Every fixed interval (RubyAuth defaults to 30 seconds), a new password is generated. This addresses several issues with traditional passwords: they can be forgotten, stolen, or guessed. OTPs solve some of these problems, but their delivery via SMS or email can be unreliable (or even risky, considering it opens new attack vectors).

TOTP, however, generates codes offline, making it both secure and convenient. You just need an authenticator app on your phone.

#### Getting TOTP URI

After enabling 2FA, you can get the TOTP URI to display to the user. This URI is generated by the server using the `secret` and `issuer` and can be used to generate a QR code for the user to scan with their authenticator app.

<APIMethod path="/two-factor/get-totp-uri" method="POST" requireSession>
  ```
  type getTOTPURI = {
      /**
       * The user password for credential accounts.
       */
      password?: string = "password123"
  }
  ```
</APIMethod>

**Example: Using React**

Once you have the TOTP URI, you can use it to generate a QR code for the user to scan with their authenticator app.

```ruby title="server.rb"
totp = auth.api.get_totp_uri(
  headers: {"cookie" => session_cookie},
  body: {password: "password123"}
)

totp[:totpURI]
```

<Callout>
  By default the issuer for TOTP is set to the app name provided in the auth config or if not provided it will be set to `RubyAuth`. You can override this by passing `issuer` to the plugin config.
</Callout>

#### Verifying TOTP

After the user has entered their 2FA code, you can verify it using `twoFactor.verifyTotp` method. `RubyAuth` follows standard practice by accepting TOTP codes from one period before and one after the current code, ensuring users can authenticate even with minor time delays on their end.

<APIMethod path="/two-factor/verify-totp" method="POST" requireHeaders headersComment="Pass the current request headers so RubyAuth can read and set the 2FA/session cookies.">
  ```
  type verifyTOTP = {
      /**
       * The TOTP code.
       */
      code: string = "123456"
  }
  ```
</APIMethod>

### OTP

OTP (One-Time Password) is similar to TOTP but a random code is generated and sent to the user's email or phone.

Before using OTP to verify the second factor, you need to configure `sendOTP` in your RubyAuth instance. This function is responsible for sending the OTP to the user's email, phone, or any other method supported by your application.

```ruby title="config/auth.rb"
require "better_auth"

auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  base_url: ENV.fetch("BETTER_AUTH_URL", "http://localhost:3000"),
  plugins: [
    BetterAuth::Plugins.two_factor(issuer: "My App")
  ]
)
```

#### Sending OTP


<APIMethod path="/two-factor/send-otp" method="POST" requireHeaders headersComment="Pass the current request headers so RubyAuth can read and set the 2FA/session cookies.">
  ```
  type sendTwoFactorOTP = {
  }
  ```
</APIMethod>

#### Verifying OTP


<APIMethod path="/two-factor/verify-otp" method="POST" requireHeaders headersComment="Pass the current request headers so RubyAuth can read and set the 2FA/session cookies.">
  ```
  type verifyTwoFactorOTP = {
      /**
       * The OTP code.
       */
      code: string = "123456"
      /**
       * Whether to trust this device.
       */
      trustDevice?: boolean = true
  }
  ```
</APIMethod>

### Backup Codes

Backup codes are generated and stored in the database. This can be used to recover access to the account if the user loses access to their phone or email.

#### Generating Backup Codes

Generate backup codes for account recovery:

<APIMethod path="/two-factor/generate-backup-codes" method="POST" requireSession>
  ```
  type generateBackupCodes = {
      /**
       * The user password for credential accounts.
       */
      password?: string = "password123"
  }
  ```
</APIMethod>

<Callout type="warn">
  When you generate backup codes, the old backup codes will be deleted and new ones will be generated.
</Callout>

#### Using Backup Codes

You can now allow users to provide a backup code as an account recovery method.

<APIMethod path="/two-factor/verify-backup-code" method="POST" requireHeaders headersComment="Pass the current request headers so RubyAuth can read and set the 2FA/session cookies.">
  ```
  type verifyBackupCode = {
      /**
       * The backup code.
       */
      code: string = "backup-code"
  }
  ```
</APIMethod>

<Callout>
  Once a backup code is used, it will be removed from the database and can't be used again.
</Callout>

#### Viewing Backup Codes

To display the backup codes to the user, you can call `viewBackupCodes` on the server. This will return the backup codes in the response. You should only do this if the user has a fresh session - a session that was just created.

<APIMethod path="/two-factor/view-backup-codes" method="POST" isServerOnly forceAsBody>
  ```
  type viewBackupCodes = {
      /**
       * The user ID.
       */
      userId: string = "user-id"
  }
  ```
</APIMethod>

### Trusted Devices

You can mark a device as trusted by passing `trustDevice` to `verifyTotp` or `verifyOtp`.

```ruby title="server.rb"
auth.api.verify_totp(
  headers: {"cookie" => two_factor_cookie},
  body: {
    code: params.fetch("code"),
    trustDevice: true
  }
)
```

When `trustDevice` is set to `true`, the current device will be remembered for 30 days. During this period, the user won't be prompted for 2FA on subsequent sign-ins from this device. The trust period is refreshed each time the user signs in successfully.

### Issuer

By adding an `issuer` you can set your application name for the 2fa application.

For example, if your user uses Google Auth, the default appName will show up as `RubyAuth`. However, by using the following code, it will show up as `my-app-name`.

```ruby title="server.rb"
auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  base_url: ENV.fetch("BETTER_AUTH_URL", "http://localhost:3000"),
  plugins: [
    BetterAuth::Plugins.two_factor(issuer: "my-app-name")
  ]
)
```

## Options

### Server

**twoFactorTable**: The name of the table that stores the two factor authentication data. Default: `twoFactor`.

**skipVerificationOnEnable**: Skip the verification process before enabling two factor for a user.

**allowPasswordless**: Allow enabling and managing 2FA without a password for users that do not have a credential account. Password is still required if a credential account exists.

**Issuer**: The issuer is the name of your application. It's used to generate TOTP codes. It'll be displayed in the authenticator apps.

### Account lockout

Account lockout limits consecutive failed second-factor verifications during sign-in. TOTP, OTP, and backup-code failures from separate challenges all increment the same per-account counter. Once the limit is reached, subsequent verification attempts—including a correct code—return HTTP `429` with `ACCOUNT_TEMPORARILY_LOCKED` until the lock expires.

```ruby title="config/auth.rb"
auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  base_url: ENV.fetch("BETTER_AUTH_URL"),
  plugins: [
    BetterAuth::Plugins.two_factor(
      issuer: "My App",
      account_lockout: {
        enabled: true,
        max_failed_attempts: 10,
        duration_seconds: 900
      },
      otp_options: {
        allowed_attempts: 5
      }
    )
  ]
)
```

export const twoFactorAccountLockoutOptionsType = {

<TypeTable type={twoFactorAccountLockoutOptionsType} />

A successful sign-in verification resets both `failedVerificationCount` and `lockedUntil`. When the lock duration elapses, the next sign-in verification clears the expired lock and starts from a zero counter. Account lockout applies only to second-factor verification during sign-in; authenticated enrollment and reverification are not blocked by it.

Choose `max_failed_attempts` and `duration_seconds` with both security and availability in mind. A very low threshold makes deliberate account lockouts easier for an attacker who can reach the second-factor step, while a high threshold permits more guesses. Setting `enabled: false` removes cross-challenge, cross-factor account protection, but it does **not** disable the fixed five-attempt TOTP/backup challenge budget or OTP's `allowed_attempts` budget.
