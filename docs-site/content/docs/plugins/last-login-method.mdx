<RubyAuthDisclaimer />

The last login method plugin tracks the most recent authentication method used by users (email, OAuth providers, etc.). This enables you to display helpful indicators on login pages, such as "Last signed in with Google" or prioritize certain login methods based on user preferences.

## Installation

<Steps>
  <Step>
    ### Add the plugin to your auth config

    ```ruby title="config/auth.rb"
    require "better_auth"

    auth = BetterAuth.auth(
      secret: ENV.fetch("BETTER_AUTH_SECRET"),
      base_url: ENV.fetch("BETTER_AUTH_URL", "http://localhost:3000"),
      plugins: [
        BetterAuth::Plugins.last_login_method
      ]
    )
    ```
  </Step>

</Steps>

## Usage

Once installed, the plugin automatically tracks the last authentication method used by users. You can then retrieve and display this information in your application.

### Getting the Last Used Method

By default, the plugin stores the most recent successful login method in the
`better-auth.last_used_login_method` cookie. Server-rendered Rack apps can read
that cookie directly; use your configured `cookie_name` if you changed it.

```ruby title="server.rb"
require "rack"

class LoginPage
  COOKIE_NAME = "better-auth.last_used_login_method"

  def call(env)
    request = Rack::Request.new(env)
    last_method = request.cookies[COOKIE_NAME]

    body = if last_method
      "Last signed in with #{last_method}"
    else
      "Choose a sign-in method"
    end

    [200, {"content-type" => "text/plain"}, [body]]
  end
end
```

### UI Integration Example

Here's how to map the cookie value to copy for a server-rendered login page:

```ruby title="server.rb"
LOGIN_METHOD_LABELS = {
  "email" => "Email and password",
  "google" => "Google",
  "magic-link" => "Magic link",
  "passkey" => "Passkey",
  "siwe" => "Ethereum wallet"
}.freeze

class LoginPage
  def call(env)
    request = Rack::Request.new(env)
    method = request.cookies["better-auth.last_used_login_method"]
    label = LOGIN_METHOD_LABELS.fetch(method, "your previous method")

    [200, {"content-type" => "text/html"}, [
      <<~HTML
        <h1>Sign in</h1>
        <p>Last signed in with #{Rack::Utils.escape_html(label)}.</p>
      HTML
    ]]
  end
end
```

## Database Persistence

By default, the last login method is stored only in cookies. For more persistent tracking and analytics, you can enable database storage.

<Steps>
  <Step>
    ### Enable database storage

    Set `store_in_database` to `true` in your plugin configuration:

    ```ruby title="config/auth.rb"
    require "better_auth"

    auth = BetterAuth.auth(
      secret: ENV.fetch("BETTER_AUTH_SECRET"),
      base_url: ENV.fetch("BETTER_AUTH_URL", "http://localhost:3000"),
      plugins: [
        BetterAuth::Plugins.last_login_method(store_in_database: true)
      ]
    )
    ```
  </Step>

  <Step>
    ### Run database migration

    The plugin will automatically add a `lastLoginMethod` field to your user table. Run the migration to apply the changes:

    <Tabs items={["migrate", "generate"]}>
  <Tab value="migrate">
    ```bash
    bundle exec better-auth migrate --cwd . --config config/better_auth.rb --yes
    ```
  </Tab>

  <Tab value="generate">
    ```bash
    bundle exec better-auth generate --cwd . --dialect postgres --output db/better_auth/schema.sql --config config/better_auth.rb
    ```
  </Tab>
</Tabs>
  </Step>

  <Step>
    ### Access database field

    When database storage is enabled, the `lastLoginMethod` field becomes available in user objects:

    ```ruby title="server.rb"
    session = auth.api.get_session(
      headers: {"cookie" => request.env.fetch("HTTP_COOKIE", "")},
      query: {disableCookieCache: true}
    )

    last_method = session&.dig(:user, "lastLoginMethod")
    ```
  </Step>
</Steps>

### Database Schema

When `store_in_database` is enabled, the plugin adds the following field to the `user` table:

Table: `user`

export const lastLoginMethodUserTableFields = [
	{
		name: "lastLoginMethod",
		type: "string",
		description: "The last authentication method used by the user",
		isOptional: true,
	},
];

<DatabaseTable name="user" fields={lastLoginMethodUserTableFields} />

### Custom Schema Configuration

You can customize the database field name:

```ruby title="config/auth.rb"
require "better_auth"

auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  base_url: ENV.fetch("BETTER_AUTH_URL", "http://localhost:3000"),
  plugins: [
    BetterAuth::Plugins.last_login_method(
      store_in_database: true,
      schema: {
        user: {
          last_login_method: "last_login_method"
        }
      }
    )
  ]
)
```

## Configuration Options

The last login method plugin accepts the following options:

### Server Options

```ruby title="config/auth.rb"
require "better_auth"

auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  base_url: ENV.fetch("BETTER_AUTH_URL", "http://localhost:3000"),
  plugins: [
    BetterAuth::Plugins.last_login_method(
      cookie_name: "my-app.last_login_method",
      max_age: 60 * 60 * 24 * 90,
      store_in_database: true,
      schema: {
        user: {
          last_login_method: "last_login_method"
        }
      }
    )
  ]
)
```

**cookie_name**: `String`

* The name of the cookie used to store the last login method
* Default: `"better-auth.last_used_login_method"`
* **Note**: This cookie is `http_only: false` so server-rendered pages can also expose the value to UI code when needed

**max_age**: `Integer`

* Cookie expiration time in seconds
* Default: `2592000` (30 days)

**store_in_database**: `Boolean`

* Whether to store the last login method in the database
* Default: `false`
* When enabled, adds a `lastLoginMethod` field to the user table

**custom_resolve_method**: `Proc`

* Custom function to determine the login method from the request context
* Return `null` to use the default resolution logic
* Useful for custom OAuth providers or authentication flows

**schema**: `object`

* Customize database field names when `store_in_database` is enabled
* Allows mapping the `lastLoginMethod` field to a custom column name

### Default Method Resolution

By default, the plugin tracks these authentication methods:

* **Email authentication**: `"email"`
* **OAuth providers**: Provider ID (e.g., `"google"`, `"github"`, `"discord"`)
* **OAuth2 callbacks**: Provider ID from URL path
* **Sign up methods**: Tracked the same as sign in methods

The plugin automatically detects the method from these endpoints:

* `/callback/:id` - OAuth callback with provider ID
* `/oauth2/callback/:id` - OAuth2 callback with provider ID
* `/sign-in/email` - Email sign in
* `/sign-up/email` - Email sign up

## Cross-Domain Support

The plugin automatically inherits cookie settings from RubyAuth's centralized cookie system. This solves the problem where the last login method wouldn't persist across:

* **Cross-subdomain setups**: `auth.example.com` → `app.example.com`
* **Cross-origin setups**: `api.company.com` → `app.different.com`

When you enable `crossSubDomainCookies` or `crossOriginCookies` in your RubyAuth config, the plugin will automatically use the same domain, secure, and sameSite settings as your session cookies, ensuring consistent behavior across your application.

## Advanced Examples

### Custom Provider Tracking

If you have custom OAuth providers or authentication methods, you can use the `custom_resolve_method` option:

```ruby title="config/auth.rb"
require "better_auth"

auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  base_url: ENV.fetch("BETTER_AUTH_URL", "http://localhost:3000"),
  plugins: [
    BetterAuth::Plugins.last_login_method(
      custom_resolve_method: ->(ctx) {
        ctx.path.start_with?("/custom-login") ? "custom" : nil
      }
    )
  ]
)
```
