<RubyAuthDisclaimer />

RubyAuth provides built-in authentication support for:

* **Email and password**
* **Social provider (Google, GitHub, Apple, and more)**

But also can easily be extended using plugins, such as: [username](/docs/plugins/username), [magic link](/docs/plugins/magic-link), [passkey](/docs/plugins/passkey), [email-otp](/docs/plugins/email-otp), and more.

<Callout type="info">
  RubyAuth does not ship an official browser client gem. Use the HTTP routes
  shown below from fetch, HTMX, Turbo, mobile clients, or server-rendered forms,
  and use `auth.api.*` from Ruby server code.
</Callout>

## Email & Password

To enable email and password authentication:

```ruby title="config/better_auth.rb"
require "better_auth"

auth = BetterAuth.auth(
  email_and_password: {
    enabled: true
  }
)
```

### Sign Up

To sign up a user you need to call the `/api/auth/sign-up/email` endpoint with the user's information.

```http
POST /api/auth/sign-up/email
Content-Type: application/json

{
  "email": "user@example.com",
  "password": "password123",
  "name": "User Name",
  "image": "https://example.com/avatar.png",
  "callbackURL": "/dashboard"
}
```

From Ruby server code, call the direct API method:

```ruby
result = auth.api.sign_up_email(
  body: {
    email: "user@example.com",
    password: "password123",
    name: "User Name",
    image: "https://example.com/avatar.png",
    callbackURL: "/dashboard"
  }
)
```

By default, the users are automatically signed in after they successfully sign up. To disable this behavior you can set `auto_sign_in` to `false`.

```ruby title="config/better_auth.rb"
require "better_auth"

auth = BetterAuth.auth(
  email_and_password: {
    enabled: true,
    auto_sign_in: false # defaults to true
  }
)
```

### Sign In

To sign a user in, call `/api/auth/sign-in/email`.

```http
POST /api/auth/sign-in/email
Content-Type: application/json

{
  "email": "user@example.com",
  "password": "password123",
  "callbackURL": "/dashboard",
  "rememberMe": false
}
```

The route returns the token, user, and a `Set-Cookie` header containing `better-auth.session_token`.

<Callout type="warn">
  Browser and mobile clients should call the HTTP endpoint. Ruby server code can
  use `auth.api.sign_in_email` directly.
</Callout>

### Server-Side Authentication

To authenticate a user on the server, you can use the `auth.api` methods.

```ruby title="server.rb"
status, headers, body = auth.api.sign_in_email(
  body: {
    email: email,
    password: password
  },
  headers: {
    "x-forwarded-for" => request.ip,
    "user-agent" => request.user_agent
  },
  as_response: true
)

cookie_header = headers.fetch("set-cookie")
```

<Callout>
  When you use `as_response: true`, RubyAuth returns a Rack-compatible response
  tuple so your framework can forward the status, headers, body, and cookies.
</Callout>

## Social Sign-On

RubyAuth supports multiple social providers, including Google, GitHub, Apple, Discord, and more. To use a social provider, you need to configure the ones you need in the `social_providers` option on your `auth` object.

```ruby title="config/better_auth.rb"
require "better_auth"

auth = BetterAuth.auth(
  social_providers: {
    github: BetterAuth::SocialProviders.github(
      client_id: ENV.fetch("GITHUB_CLIENT_ID"),
      client_secret: ENV.fetch("GITHUB_CLIENT_SECRET")
    )
  }
)
```

### Sign in with social providers

To sign in using a social provider you need to call `/api/auth/sign-in/social`. It takes an object with the following properties:

```http
POST /api/auth/sign-in/social
Content-Type: application/json

{
  "provider": "github",
  "callbackURL": "/dashboard",
  "errorCallbackURL": "/error",
  "newUserCallbackURL": "/welcome",
  "disableRedirect": true
}
```

From Ruby:

```ruby
response = auth.api.sign_in_social(
  body: {
    provider: "github",
    callback_url: "/dashboard",
    error_callback_url: "/error",
    new_user_callback_url: "/welcome",
    disable_redirect: true
  }
)

redirect_url = response.fetch(:url)
```

You can also authenticate using `idToken` or `accessToken` from the social provider instead of redirecting the user to the provider's site. See social providers documentation for more details.

## Signout

To signout a user, call `/api/auth/sign-out` with the user's session cookie.

```http
POST /api/auth/sign-out
Cookie: better-auth.session_token=...
```

From Ruby:

```ruby
status, headers, body = auth.api.sign_out(
  headers: {"cookie" => cookie_header},
  as_response: true
)
```

You can redirect to your login page after the response succeeds.

## Session

Once a user is signed in, you'll want to access the user session. RubyAuth allows you to easily access the session data from both the server and client sides.

### Client Side

#### Use Session

RubyAuth does not provide a framework-specific `useSession` hook. Use `/api/auth/get-session` and store the returned session in your frontend state.

```http
GET /api/auth/get-session
Cookie: better-auth.session_token=...
```

#### Get Session

If you prefer not to maintain local state, call the same `get-session` endpoint whenever you need the current session.

```http
GET /api/auth/get-session?disableCookieCache=true
Cookie: better-auth.session_token=...
```

You can also use it with client-side data-fetching libraries like [TanStack Query](https://tanstack.com/query/latest).

### Server Side

The server provides a `session` object that you can use to access the session data. It requires request headers object to be passed to the `getSession` method.

**Example: Using Ruby frameworks**

<Tabs items={["Rails", "Rack"]}>
  <Tab value="Rails">
    ```ruby title="app/controllers/application_controller.rb"
    class ApplicationController < ActionController::Base
      private

      def current_auth_session
        auth.api.get_session(
          headers: {"cookie" => request.headers["Cookie"].to_s}
        )
      end
    end
    ```
  </Tab>

  <Tab value="Rack">
    ```ruby title="app.rb"
    request = Rack::Request.new(env)

    session = auth.api.get_session(
      headers: {"cookie" => request.get_header("HTTP_COOKIE").to_s}
    )
    ```
  </Tab>
</Tabs>

<Callout>
  For more details check [session-management](/docs/concepts/session-management) documentation.
</Callout>

## Using Plugins

One of the unique features of RubyAuth is a plugins ecosystem. It allows you to add complex auth related functionality with small lines of code.

Below is an example of how to add two factor authentication using two factor plugin.

<Steps>
  <Step>
    ### Server Configuration

    To add a plugin, you need to import the plugin and pass it to the `plugins` option of the auth instance. For example, to add two factor authentication, you can use the following code:

    ```ruby title="config/better_auth.rb"
    require "better_auth"

    auth = BetterAuth.auth(
      # ... rest of the options
      plugins: [
        BetterAuth::Plugins.two_factor
      ]
    )
    ```

    now two factor related routes and method will be available on the server.
  </Step>

  <Step>
    ### Migrate Database

    After adding the plugin, you'll need to add the required tables to your database. You can do this by running the `migrate` command, or by using the `generate` command to create the schema and handle the migration manually.

    generating the schema:

    ```bash title="terminal"
    bundle exec better-auth generate --cwd . --dialect postgres --output db/better_auth/schema.sql --config config/better_auth.rb
    ```

    using the `migrate` command:

    ```bash title="terminal"
    bundle exec better-auth migrate --cwd . --config config/better_auth.rb --yes
    ```

    <Callout>
      If you prefer adding the schema manually, you can check the schema required on the [two factor plugin](/docs/plugins/2fa#schema) documentation.
    </Callout>
  </Step>

  <Step>
    ### Client Configuration

    Once we're done with the server, browser and app clients can call the plugin's HTTP endpoints. RubyAuth does not ship a JavaScript two-factor client plugin.

    ```http
    POST /api/auth/two-factor/enable
    Content-Type: application/json
    Cookie: better-auth.session_token=...

    {"password":"password123"}
    ```

    Server-side Ruby code can call the plugin API methods directly when they are enabled by the plugin:

    ```ruby
    auth.api.enable_two_factor(
      body: {password: "password123"},
      headers: {"cookie" => cookie_header}
    )
    ```
  </Step>

  <Step>
    Next step: See the <Link href="/docs/plugins/2fa">two factor plugin documentation</Link>.
  </Step>
</Steps>
