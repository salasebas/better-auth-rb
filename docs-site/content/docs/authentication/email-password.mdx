<RubyAuthDisclaimer />

Email and password authentication is a common method used by many applications. RubyAuth provides a built-in email and password authenticator that you can easily integrate into your project.

<Callout type="info">
  If you prefer username-based authentication, check out the{" "}
  <Link href="/docs/plugins/username">username plugin</Link>. It extends the
  email and password authenticator with username support.
</Callout>

## Enable Email and Password

To enable email and password authentication, you need to set the `email_and_password.enabled` option to `true` in the `auth` configuration.

<Tabs items={["Rails", "Plain Ruby"]}>
  <Tab value="Rails">
    ```ruby title="config/initializers/better_auth.rb"
    BetterAuth::Rails.configure do |config|
      config.email_and_password do |email_and_password|
        email_and_password.enabled = true
      end
    end
    ```
  </Tab>

  <Tab value="Plain Ruby">
    ```ruby title="config/better_auth.rb"
    require "better_auth"

    auth = BetterAuth.auth(
      email_and_password: {
        enabled: true
      }
    )
    ```
  </Tab>
</Tabs>

<Callout type="info">
  If it's not enabled, it'll not allow you to sign in or sign up with email and
  password.
</Callout>

Additionally, to use the following server methods like `sign_up_email`, cookies must also be passed back to the client. Use `as_response: true` when you need a Rack-compatible status, headers, and body response from Ruby server code.

## Usage

### Sign Up

To sign a user up, call the email sign-up endpoint.

<APIMethod path="/sign-up/email" method="POST">
  ```http
  POST /api/auth/sign-up/email
  Content-Type: application/json

  {
    "name": "John Doe",
    "email": "john.doe@example.com",
    "password": "password1234",
    "image": "https://example.com/image.png",
    "callbackURL": "https://example.com/callback"
  }
  ```
</APIMethod>

From Ruby:

```ruby
result = auth.api.sign_up_email(
  body: {
    name: "John Doe",
    email: "john.doe@example.com",
    password: "password1234",
    image: "https://example.com/image.png",
    callbackURL: "https://example.com/callback"
  }
)
```

When you need to forward cookies from a Ruby controller or Rack endpoint, request a Rack-compatible response tuple:

```ruby
status, headers, body = auth.api.sign_up_email(
  body: {
    name: "John Doe",
    email: "john.doe@example.com",
    password: "password1234"
  },
  as_response: true
)
```

<Callout>
  These are the default properties for the sign up email endpoint, however it's
  possible that with [additional fields](/docs/concepts/database#extending-core-schema)
  or special plugins you can pass more properties to the endpoint.
</Callout>

### Sign In

To sign a user in, call the email sign-in endpoint.

<APIMethod path="/sign-in/email" method="POST" requireSession>
  ```http
  POST /api/auth/sign-in/email
  Content-Type: application/json

  {
    "email": "john.doe@example.com",
    "password": "password1234",
    "rememberMe": true,
    "callbackURL": "https://example.com/callback"
  }
  ```
</APIMethod>

From Ruby:

```ruby
status, headers, body = auth.api.sign_in_email(
  body: {
    email: "john.doe@example.com",
    password: "password1234",
    rememberMe: true,
    callbackURL: "https://example.com/callback"
  },
  as_response: true
)
```

<Callout>
  These are the default properties for the sign in email endpoint, however it's
  possible that with [additional fields](/docs/concepts/database#extending-core-schema)
  or special plugins you can pass different properties to the endpoint.
</Callout>

### Sign Out

To sign a user out, call the sign-out endpoint with the user's session cookie.

<APIMethod path="/sign-out" method="POST" requireSession noResult>
  ```http
  POST /api/auth/sign-out
  Cookie: better-auth.session_token=...
  ```
</APIMethod>

You can redirect to your login page after the response succeeds.

```ruby
status, headers, body = auth.api.sign_out(
  headers: {"cookie" => cookie_header},
  as_response: true
)
```

### Email Verification

To enable email verification, you need to pass a function that sends a verification email with a link. The `send_verification_email` function takes a data object with the following properties:

* `user`: The user object.
* `url`: The URL to send to the user which contains the token.
* `token`: A verification token used to complete the email verification.

and a `request` object as the second parameter.

```ruby title="config/better_auth.rb"
require "better_auth"

auth = BetterAuth.auth(
  email_verification: {
    send_verification_email: lambda do |data, _request = nil|
      Mailer.verify_email(
        to: data[:user]["email"],
        subject: "Verify your email address",
        text: "Click the link to verify your email: #{data[:url]}"
      ).deliver_later
    end
  }
)
```

<Callout type="warn">
  Avoid blocking the request on slow email delivery to reduce timing attack
  risk. Queue the email or hand it to your background job system.
</Callout>

HTTP clients can call `/api/auth/send-verification-email` to send a verification link to a user. This will trigger the `send_verification_email` function you provided in the `auth` configuration.

Once the user clicks on the link in the email, if the token is valid, the user will be redirected to the URL provided in the `callbackURL` parameter. If the token is invalid, the user will be redirected to the URL provided in the `callbackURL` parameter with an error message in the query string `?error=invalid_token`.

#### Require Email Verification

If you enable require email verification, users must verify their email before they can log in. And every time a user tries to sign in, `send_verification_email` is called.

<Callout>
  This only works if you have `send_verification_email` implemented and if the
  user is trying to sign in with email and password.

  When `require_email_verification` is enabled, signing up with an existing email returns a success response instead of an error to prevent user enumeration.
</Callout>

```ruby title="config/better_auth.rb"
auth = BetterAuth.auth(
  email_and_password: {
    enabled: true,
    require_email_verification: true
  },
  email_verification: {
    send_verification_email: ->(data, _request = nil) {
      Mailer.verify_email(to: data[:user]["email"], url: data[:url]).deliver_later
    }
  }
)
```

You can use the `on_existing_user_sign_up` callback to notify the existing user when someone tries to register with their email address:

```ruby title="config/better_auth.rb"
auth = BetterAuth.auth(
  email_and_password: {
    enabled: true,
    require_email_verification: true,
    on_existing_user_sign_up: lambda do |data, _request = nil|
      Mailer.security_notice(
        to: data[:user]["email"],
        subject: "Sign-up attempt with your email",
        text: "Someone tried to create an account using your email address."
      ).deliver_later
    end
  }
)
```

If a user tries to sign in without verifying their email, the sign-in endpoint returns a `403` error with the upstream `EMAIL_NOT_VERIFIED` message. Show a verification prompt and offer to send another verification email.

```http
POST /api/auth/send-verification-email
Content-Type: application/json

{
  "email": "email@example.com",
  "callbackURL": "/dashboard"
}
```

#### Email Enumeration Protection

When `require_email_verification` is enabled or `auto_sign_in` is set to `false`, the sign-up endpoint prevents email enumeration by returning the same `200` response whether the email is already registered or not. This follows [OWASP authentication best practices](https://cheatsheetseries.owasp.org/cheatsheets/Authentication_Cheat_Sheet.html#authentication-and-error-messages).

<Callout type="info">
  This protection is only active when the sign-up response does not include a
  session token - i.e., when `require_email_verification` is `true` or
  `auto_sign_in` is `false`. With the default configuration, the endpoint still
  returns a `422` error for existing emails.
</Callout>

Similarly, the `/change-email` endpoint no longer reveals whether the target email is already registered - it always returns a success response.

##### Plugins that add user fields

If you use plugins that add fields to the user table (e.g. [admin](/docs/plugins/admin), [two-factor](/docs/plugins/2fa), [phone-number](/docs/plugins/phone-number)), the synthetic response needs to include those fields to be indistinguishable from a real sign-up. Use the `custom_synthetic_user` option to build the complete user object:

```ruby title="config/better_auth.rb"
auth = BetterAuth.auth(
  email_and_password: {
    enabled: true,
    require_email_verification: true,
    custom_synthetic_user: lambda do |data|
      data[:core_fields]
        .merge(
          "role" => "user",
          "banned" => false,
          "banReason" => nil,
          "banExpires" => nil
        )
        .merge(data[:additional_fields])
        .merge("id" => data[:id])
    end
  },
  plugins: [BetterAuth::Plugins.admin]
)
```

The callback receives three building blocks:

* **`core_fields`** - `name`, `email`, `emailVerified`, `image`, `createdAt`, `updatedAt`
* **`additional_fields`** - Your `user.additional_fields` with defaults applied
* **`id`** - A generated user ID matching your configured ID strategy

You assemble them in the same order as your database schema: core fields -> plugin fields -> additional fields -> id. Each plugin documents the fields you need to add - see [admin plugin](/docs/plugins/admin#email-enumeration-protection) for an example.

#### Triggering manually Email Verification

You can trigger the email verification manually by calling the `sendVerificationEmail` endpoint.

```http
POST /api/auth/send-verification-email
Content-Type: application/json

{
  "email": "user@email.com",
  "callbackURL": "/"
}
```

```ruby
auth.api.send_verification_email(
  body: {
    email: "user@email.com",
    callbackURL: "/"
  }
)
```

### Request Password Reset

To allow users to reset a password first you need to provide `send_reset_password` function to the email and password authenticator. The `send_reset_password` function takes a data object with the following properties:

* `user`: The user object.
* `url`: The URL to send to the user which contains the token.
* `token`: A verification token used to complete the password reset.

and a `request` object as the second parameter.

```ruby title="config/better_auth.rb"
require "better_auth"

auth = BetterAuth.auth(
  email_and_password: {
    enabled: true,
    send_reset_password: lambda do |data, _request = nil|
      Mailer.reset_password(
        to: data[:user]["email"],
        subject: "Reset your password",
        text: "Click the link to reset your password: #{data[:url]}"
      ).deliver_later
    end,
    on_password_reset: lambda do |data, _request = nil|
      Rails.logger.info("Password for user #{data[:user]["email"]} has been reset.")
    end
  }
)
```

<Callout type="warn">
  Avoid blocking the request on slow email delivery to reduce timing attack
  risk. Queue the email or hand it to your background job system.
</Callout>

Additionally, you can provide an `on_password_reset` callback to execute logic after a password has been successfully reset.

Once you configured your server you can call `/api/auth/request-password-reset` to send a reset password link to the user. If the user exists, it will trigger the `send_reset_password` function you provided in the auth config.

<APIMethod path="/request-password-reset" method="POST">
  ```http
  POST /api/auth/request-password-reset
  Content-Type: application/json

  {
    "email": "john.doe@example.com",
    "redirectTo": "https://example.com/reset-password"
  }
  ```
</APIMethod>

From Ruby:

```ruby
auth.api.request_password_reset(
  body: {
    email: "john.doe@example.com",
    redirectTo: "https://example.com/reset-password"
  }
)
```

When a user clicks on the link in the email, they will be redirected to the reset password page. You can add the reset password page to your app. Then call `/api/auth/reset-password` or `auth.api.reset_password` to reset the password. It takes an object with the following properties:

* `newPassword`: The new password of the user.

```http
POST /api/auth/reset-password
Content-Type: application/json

{
  "newPassword": "password1234",
  "token": "reset-token"
}
```

<APIMethod path="/reset-password" method="POST">
  ```ruby
  auth.api.reset_password(
    body: {
      newPassword: "password1234",
      token: token
    }
  )
  ```
</APIMethod>

### Update password

A user's password isn't stored in the user table. Instead, it's stored in the account table. To change the password of a user, you can use one of the following approaches:

<APIMethod path="/change-password" method="POST" requireSession>
  ```http
  POST /api/auth/change-password
  Content-Type: application/json
  Cookie: better-auth.session_token=...

  {
    "newPassword": "newpassword1234",
    "currentPassword": "oldpassword1234",
    "revokeOtherSessions": true
  }
  ```
</APIMethod>

From Ruby:

```ruby
auth.api.change_password(
  body: {
    newPassword: "newpassword1234",
    currentPassword: "oldpassword1234",
    revokeOtherSessions: true
  },
  headers: {"cookie" => cookie_header}
)
```

### Configuration

**Password**

RubyAuth stores passwords inside the `account` table with `providerId` set to `credential`.

**Password Hashing**: RubyAuth uses `scrypt` to hash passwords by default. The `scrypt` algorithm is designed to be slow and memory-intensive to make it difficult for attackers to brute force passwords. RubyAuth can also use bcrypt when the optional `bcrypt` gem is installed.

Use bcrypt by setting the top-level `password_hasher` option:

```ruby title="Gemfile"
gem "bcrypt"
```

```ruby title="config/better_auth.rb"
auth = BetterAuth.auth(
  email_and_password: {
    enabled: true
  },
  password_hasher: :bcrypt
)
```

You can pass custom password hashing functions by setting the `password` option in the `email_and_password` configuration.

**Example**

Here's an example of customizing the password hashing:

```ruby title="config/better_auth.rb"
auth = BetterAuth.auth(
  email_and_password: {
    enabled: true,
    password: {
      hash: ->(password) { MyPasswordHasher.hash(password) },
      verify: ->(password:, hash:) { MyPasswordHasher.verify(password, hash) }
    }
  }
)
```

| Option | Type | Default | Description |
|--------|------|---------|-------------|
| `enabled` | boolean | `false` | Enable email and password authentication. |
| `disable_sign_up` | boolean | `false` | Disable email and password sign up. |
| `min_password_length` | number | `8` | The minimum length of a password. |
| `max_password_length` | number | `128` | The maximum length of a password. |
| `send_reset_password` | function | - | Sends a password reset email. It takes a function that receives reset token and user data. |
| `on_password_reset` | function | - | A callback function that is triggered when a user's password is changed successfully. |
| `on_existing_user_sign_up` | function | `nil` | A callback triggered when someone signs up with an already-registered email. Only called when enumeration protection is active (`require_email_verification: true` or `auto_sign_in: false`). |
| `custom_synthetic_user` | function | - | Build a custom synthetic user for email enumeration protection. Use when plugins add fields to the user table to ensure the fake response is indistinguishable from a real sign-up. |
| `auto_sign_in` | boolean | `true` | Automatically sign in the user after sign up. When set to false, the sign-up response returns a success response and enables enumeration protection. |
| `require_email_verification` | boolean | `false` | Require users to verify their email before they can sign in. When enabled, the sign-up response returns a success response and enables enumeration protection. |
| `revoke_sessions_on_password_reset` | boolean | `false` | Whether to revoke all other sessions when resetting password. |
| `reset_password_token_expires_in` | number | `3600` | Number of seconds the reset password token is valid for. |
| `password.hash` | function | - | Custom password hashing function. |
| `password.verify` | function | - | Custom password verification function. |
