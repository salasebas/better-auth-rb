PostgreSQL is a powerful, open-source relational database management system known for its advanced features, extensibility, and support for complex queries and large datasets.
Read more [here](https://www.postgresql.org/).

<Callout type="info">
Use the lambda form when configuring the direct adapter. Better Auth passes the
final configuration into the lambda, so the adapter sees plugins, custom model
names, custom field names, and `advanced.database.generate_id`.
</Callout>

## Install Dependencies

```ruby title="Gemfile"
gem "better_auth"
gem "pg"
```

```bash title="Terminal"
bundle install
```

## Example Usage

Make sure you have PostgreSQL installed and configured.
Then, you can connect it straight into Better Auth.

```ruby title="config/auth.rb"
require "better_auth"

auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  base_url: ENV.fetch("BETTER_AUTH_URL"),
  database: ->(options) {
    BetterAuth::Adapters::Postgres.new(
      options,
      url: ENV.fetch("DATABASE_URL")
    )
  }
)
```

<Callout>
  For more information, read the Ruby
  `BetterAuth::Adapters::Postgres` implementation and adapter tests in this repository.
</Callout>

You may also pass an existing `PG::Connection` when your app owns connection
management.

```ruby title="config/auth.rb"
require "pg"

connection = PG.connect(ENV.fetch("DATABASE_URL"))

auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  database: ->(options) {
    BetterAuth::Adapters::Postgres.new(
      options,
      connection: connection
    )
  }
)
```

## Rails

Rails apps should normally use `better_auth-rails` and the ActiveRecord adapter.

```ruby title="Gemfile"
gem "better_auth-rails"
gem "pg"
```

```bash title="Terminal"
bin/rails generate better_auth:install
bin/rails db:migrate
```

Rails uses ActiveRecord by default:

```ruby title="config/initializers/better_auth.rb"
BetterAuth::Rails.configure do |config|
  config.database_adapter = :active_record
end
```

## Joins (Experimental)

Database joins is useful when Better-Auth needs to fetch related data from multiple tables in a single query.
Endpoints like `/get-session`, `/get-full-organization` and many others benefit greatly from this feature,
seeing upwards of 2x to 3x performance improvements depending on database latency.

The Ruby PostgreSQL adapter supports joins.
To enable this feature, you need to set the `experimental.joins` option to `true` in your auth configuration.

```ruby title="config/auth.rb"
auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  experimental: { joins: true },
  database: ->(options) {
    BetterAuth::Adapters::Postgres.new(
      options,
      url: ENV.fetch("DATABASE_URL")
    )
  }
)
```

<Callout type="warn">
  It's possible that you may need to run migrations after enabling this feature.
</Callout>

## Use a non-default schema

In most cases, the default schema is `public`. To have Better Auth use a
non-default schema (e.g., `auth`) for its tables, you have several options:

### Option 1: Set search\_path in connection string (Recommended)

Append the `options` parameter to your connection URI:

```ruby title="config/auth.rb"
auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  base_url: ENV.fetch("BETTER_AUTH_URL"),
  database: ->(options) {
    BetterAuth::Adapters::Postgres.new(
      options,
      url: "postgres://user:password@localhost:5432/database?options=-c search_path=auth"
    )
  }
)
```

URL-encode if needed: `?options=-c%20search_path%3Dauth`.

### Option 2: Set search\_path using Pool options

The Ruby adapter accepts an existing `PG::Connection`, so applications that own
connection setup can pass a connection initialized with the same PostgreSQL
runtime option:

```ruby title="config/auth.rb"
require "pg"

connection = PG.connect(
  host: "localhost",
  port: 5432,
  user: "postgres",
  password: "password",
  dbname: "my-db",
  options: "-c search_path=auth"
)

auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  base_url: ENV.fetch("BETTER_AUTH_URL"),
  database: ->(options) {
    BetterAuth::Adapters::Postgres.new(options, connection: connection)
  }
)
```

### Option 3: Set default schema for database user

Set the PostgreSQL user's default schema:

```sql
ALTER USER your_user SET search_path TO auth;
```

After setting this, reconnect to apply the changes.

### Prerequisites

Before using a non-default schema, ensure:

1. **The schema exists:**
   ```sql
   CREATE SCHEMA IF NOT EXISTS auth;
   ```

2. **The user has appropriate permissions:**
   ```sql
   GRANT ALL PRIVILEGES ON SCHEMA auth TO your_user;
   GRANT ALL PRIVILEGES ON ALL TABLES IN SCHEMA auth TO your_user;
   ALTER DEFAULT PRIVILEGES IN SCHEMA auth GRANT ALL ON TABLES TO your_user;
   ```

### How it works

The Better Auth CLI migration system automatically detects your configured `search_path`:

* When running `bundle exec better-auth migrate`, it inspects only the tables in your configured schema
* Tables in other schemas (e.g., `public`) are ignored, preventing conflicts
* All new tables are created in your specified schema

### Troubleshooting

<Callout type="warning">
  **Issue:** "relation does not exist" error during migration

  **Solution:** This usually means the schema doesn't exist or the user lacks permissions. Create the schema and grant permissions as shown above.
</Callout>

<Callout type="info">
  **Verifying your schema configuration:**

  You can verify which schema Better Auth will use by checking the `search_path`:

  ```sql
  SHOW search_path;
  ```

  This should return your custom schema (e.g., `auth`) as the first value.
</Callout>

## Schema

The Ruby schema maps Better Auth logical fields to PostgreSQL physical columns.

| Better Auth field | PostgreSQL column |
| --- | --- |
| `emailVerified` | `email_verified` |
| `userId` | `user_id` |
| `createdAt` | `created_at` |
| `expiresAt` | `expires_at` |

JSON-like fields use `jsonb`, date fields use `timestamptz`, and ids are
generated as hex strings by default. Set
`advanced: { database: { generate_id: "uuid" } }` to generate UUID strings
instead.

## Additional Information

PostgreSQL is supported under the hood via `BetterAuth::Adapters::Postgres`. Other relational database options are documented in <Link href="/docs/adapters/other-relational-databases">Other Relational Databases</Link>.

If you're looking for performance improvements or tips, take a look at our guide to <Link href="/docs/guides/optimizing-for-performance">performance optimizations</Link>.

- The direct adapter supports the shared Better Auth CRUD contract, including
  operators, sorting, pagination, counts, transactions, and native joins.
- Rails apps should prefer Rails migrations generated by `better_auth-rails`;
  direct SQL generation is intended for non-Rails apps or custom migration
  workflows.
