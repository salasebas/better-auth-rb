<RubyAuthDisclaimer />

Microsoft SQL Server is a relational database management system developed by Microsoft, designed for enterprise-level data storage, management, and analytics with robust security and scalability features.
Read more [here](https://en.wikipedia.org/wiki/Microsoft_SQL_Server).

<Callout type="info">
MSSQL support uses Sequel internally for safer parameter binding over TinyTDS.
This does not expose a public Sequel adapter; configure
`BetterAuth::Adapters::MSSQL` directly with a URL or an existing Sequel
database object.
</Callout>

## Install Dependencies

```ruby title="Gemfile"
gem "better_auth"
gem "sequel"
gem "tiny_tds"
```

```bash title="Terminal"
bundle install
```

## Example Usage

Make sure you have MS SQL installed and configured.
Then, you can connect it straight into Better Auth.

```ruby title="config/auth.rb"
require "better_auth"

auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  base_url: ENV.fetch("BETTER_AUTH_URL"),
  database: ->(options) {
    BetterAuth::Adapters::MSSQL.new(
      options,
      url: ENV.fetch("DATABASE_URL")
    )
  }
)
```

<Callout>
  For more information, read the Ruby
  `BetterAuth::Adapters::MSSQL` implementation and adapter tests in this repository.
</Callout>

You may also pass an existing Sequel database object when your app owns
connection management.

```ruby title="config/auth.rb"
require "sequel"
require "tiny_tds"

database = Sequel.connect(ENV.fetch("DATABASE_URL"))

auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  database: ->(options) {
    BetterAuth::Adapters::MSSQL.new(
      options,
      connection: database
    )
  }
)
```

## Joins (Experimental)

Database joins is useful when Better-Auth needs to fetch related data from multiple tables in a single query.
Endpoints like `/get-session`, `/get-full-organization` and many others benefit greatly from this feature,
seeing upwards of 2x to 3x performance improvements depending on database latency.

The Ruby MS SQL adapter supports joins.
To enable this feature, you need to set the `experimental.joins` option to `true` in your auth configuration.

```ruby title="config/auth.rb"
auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  experimental: { joins: true },
  database: ->(options) {
    BetterAuth::Adapters::MSSQL.new(
      options,
      url: ENV.fetch("DATABASE_URL")
    )
  }
)
```

<Callout type="warn">
  It's possible that you may need to run migrations after enabling this feature.
</Callout>

## Schema

The Ruby schema maps Better Auth logical fields to MSSQL physical columns.

| Better Auth field | MSSQL column |
| --- | --- |
| `emailVerified` | `email_verified` |
| `userId` | `user_id` |
| `createdAt` | `created_at` |
| `expiresAt` | `expires_at` |

JSON-like fields are stored as `varchar(8000)` JSON strings and parsed on
output, date fields use `datetime2(3)`, indexed strings use `varchar(255)`, and
ids are generated as hex strings by default. Set
`advanced: { database: { generate_id: "uuid" } }` to generate UUID strings
instead.

## Additional Information

MS SQL is supported under the hood via `BetterAuth::Adapters::MSSQL`. Other relational database options are documented in <Link href="/docs/adapters/other-relational-databases">Other Relational Databases</Link>.

If you're looking for performance improvements or tips, take a look at our guide to <Link href="/docs/guides/optimizing-for-performance">performance optimizations</Link>.

- The direct adapter supports the shared Better Auth CRUD contract, including
  operators, sorting, pagination, counts, Sequel-backed transactions, and native
  joins.
- The `DATABASE_URL` should use a Sequel/TinyTDS-compatible connection URL.
- Direct SQL generation is intended for non-Rails apps or custom migration
  workflows.
