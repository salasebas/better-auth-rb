## Introduction

In this guide, we'll walk through the steps to migrate a project from [Auth.js](https://authjs.dev/) (formerly [NextAuth.js](https://next-auth.js.org/)) to RubyAuth. Since these projects have different design philosophies, the migration requires careful planning and work. If your current setup is working well, there's no urgent need to migrate. We continue to handle security patches and critical issues for Auth.js.

However, if you're starting a new project or facing challenges with your current setup, we strongly recommend using RubyAuth. Our roadmap includes features previously exclusive to Auth.js, and we hope this will unite the ecosystem more strongly without causing fragmentation.

      <Tab value="RubyAuth">

        ```ruby
        auth = BetterAuth.auth(
          secret: ENV.fetch("BETTER_AUTH_SECRET"),

          }
        )
        ```

  </Step>

    ## Replace client SDK calls

    Auth.js client helpers cannot be reused directly from RubyAuth. Replace them with Rails/Rack helpers for server-rendered pages or HTTP calls to the mounted RubyAuth endpoints from browser code.

    <UnderDevelopment>
      RubyAuth does not ship a JavaScript client SDK. Keep this step as an integration boundary and avoid copying Auth.js client code into the Ruby app.
    </UnderDevelopment>
  </Step>

    ## Protecting Resources

    > Proxy(Middleware) is not intended for slow data fetching. While Proxy can be helpful for optimistic checks such as permission-based redirects, it should not be used as a full session management or authorization solution. - [Next.js docs](https://nextjs.org/docs/app/getting-started/proxy#use-cases)

    Auth.js offers approaches using Proxy (Middleware), but we recommend handling auth checks on each page or route rather than in a Proxy or Layout. Here is a basic example of protecting resources with RubyAuth.

    <Tabs items={["Client-side", "Server-side"]}>
      <Tab value="Client-side">
        ```ruby title="app/dashboard/page.rb"
        # Rails controller/helper example
        before_action :require_current_user

        def require_current_user
          @session = BetterAuth::Rails.current_session(request)
          redirect_to "/sign-in" unless @session
        end
        ```
      </Tab>

      <Tab value="Server-side">
        ```ruby title="app/dashboard/page.rb"

          redirect_to "/sign-in" unless @session
        end
        ```

    ### Comparison

    Table: <strong className="underline italic">User</strong>

    * `name`, `email`, and `email_verified` are required in RubyAuth, while optional in Auth.js
    * `email_verified` uses a boolean in RubyAuth, while Auth.js uses a timestamp
    * RubyAuth includes `created_at` and `updated_at` timestamps

    Table: <strong className="underline italic">Session</strong>

    * RubyAuth uses `token` instead of `session_token`
    * RubyAuth uses `expires_at` instead of `expires`
    * RubyAuth includes `ip_address` and `userAgent` fields
    * RubyAuth includes `created_at` and `updated_at` timestamps

    Table: <strong className="underline italic">Account</strong>

    * RubyAuth uses camelCase naming (e.g. `refresh_token` vs `refresh_token`)
    * RubyAuth includes `accountId` to distinguish between the account ID and internal ID
    * RubyAuth uses `provider_id` instead of `provider`
    * RubyAuth includes `access_tokenExpiresAt` and `refresh_tokenExpiresAt` for token management
    * RubyAuth includes `password` field to support built-in credential authentication
    * RubyAuth does not have a `type` field as it's determined by the `provider_id`
    * RubyAuth removes `token_type` and `session_state` fields
    * RubyAuth includes `created_at` and `updated_at` timestamps

    <Callout type="warn">
      **Critical for Credentials Migration:**
      In Auth.js, passwords are often stored directly on the `User` model. In RubyAuth, passwords must be stored in the `Account` table.
      If you are migrating users with passwords (including for the Phone Number plugin), you **must** create a record in the `account` table for each user with the `provider_id` set specifically to `"credential"`. Without this record, password-based sign-ins will fail.
    </Callout>

    Table: <strong className="underline italic">VerificationToken -> Verification</strong>

    * RubyAuth uses `Verification` table instead of `VerificationToken`
    * RubyAuth uses a single `id` primary key instead of composite primary key
    * RubyAuth uses `value` instead of `token` to support various verification types
    * RubyAuth uses `expires_at` instead of `expires`
    * RubyAuth includes `created_at` and `updated_at` timestamps

    <Callout type="info">
      If you were using Auth.js v4, note that v5 does not introduce any breaking changes to the database schema. Optional fields like `oauth_token_secret` and `oauth_token` can be removed if you are not using them. Rarely used fields like `refresh_token_expires_in` can also be removed.
    </Callout>

    ### Customization

    You may have extended the database models or implemented additional logic in Auth.js. RubyAuth allows you to customize the core schema in a type-safe way. You can also define custom logic during the lifecycle of database operations. For more details, see [Concepts - Database](/docs/concepts/database).
