---
title: Multiple Serving Origins
description: Serve RubyAuth from approved hosts while preserving one canonical identity.
---

<RubyAuthDisclaimer />

RubyAuth separates its canonical identity from the origins that may serve a request:

* `base_url` is required (or supplied by `BETTER_AUTH_URL`) and is the stable canonical URL used for OAuth/OIDC identity, provider callback defaults, JWT issuer/audience, and OpenAPI metadata.
* `serving_origins` is an optional allowlist of full origin patterns that may be selected from an inbound request for self-referential links such as email verification, password reset, account deletion, and magic links.

An unrecognized request host safely falls back to `base_url`. RubyAuth never derives its canonical identity from `Host` or forwarded headers.

## Basic setup

```ruby title="config/initializers/better_auth.rb"
auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  base_url: ENV.fetch("BETTER_AUTH_URL"),
  serving_origins: [
    "https://myapp.com",
    "https://www.myapp.com",
    "https://*.vercel.app"
  ],
  database: :postgres
)
```

`serving_origins` entries must include `http://` or `https://` and must not include a path. The existing wildcard matching rules apply. The canonical origin is always a serving origin, even when omitted from the list.

## Reverse proxies

RubyAuth reads `Host` for the request-facing candidate. It reads `X-Forwarded-Host` and `X-Forwarded-Proto` only when proxy trust is explicitly enabled:

```ruby
auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  base_url: "https://auth.example.com",
  serving_origins: ["https://*.example.com"],
  advanced: { trusted_proxy_headers: true }
)
```

Only enable proxy-header trust when your edge proxy overwrites those headers and clients cannot supply them directly. A syntactically valid header is still ignored unless its full origin matches `serving_origins`.

## Serving origins versus trusted origins

Every serving origin is automatically trusted for origin and redirect checks. The reverse is intentionally false: adding a browser frontend to `trusted_origins` does not authorize that frontend's host to receive token-bearing links.

```ruby
auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  base_url: "https://auth.example.com",
  serving_origins: ["https://tenant.example.com"],
  trusted_origins: ["https://frontend.example.com"]
)
```

Here `tenant.example.com` may serve auth routes and appears in generated capability links. `frontend.example.com` may pass CSRF/redirect validation, but can never become the auth serving origin.

## Migrating from dynamic `base_url`

The former hash form of `base_url` is rejected. Replace its fallback with the canonical `base_url` and convert allowed hosts to full `serving_origins`:

```ruby
# Before
base_url: {
  allowed_hosts: ["app.example.com", "*.vercel.app"],
  protocol: "https",
  fallback: "https://app.example.com"
}

# After
base_url: "https://app.example.com",
serving_origins: ["https://app.example.com", "https://*.vercel.app"]
```

There is no request-inferred fallback and no first-request caching. Each request gets isolated serving state, and state is cleared after both Rack and direct `auth.api` calls.

## Token links

Password-reset, email-verification, account-deletion, and magic-link URLs use only the canonical URL or a request origin explicitly matched by `serving_origins`. An unrecognized request host falls back to the canonical URL and can never control a token-bearing link.

Cross-subdomain cookie domains remain explicit under `advanced.cross_subdomain_cookies.domain`; they are not derived from the request host.
