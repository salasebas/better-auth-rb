In this guide, we’ll walk through how to migrate a project from WorkOS to RubyAuth, covering how to move a basic WorkOS setup integrated with a Next.js app and the key considerations to keep in mind.

## Before we begin

Before getting started, let’s review which WorkOS authentication features are fully or partially supported in RubyAuth. If a feature you use in WorkOS is available via a plugin, you’ll need to configure it in the next step.

  <Step>
    ## Create RubyAuth Instance

    First, set up RubyAuth in your project. Follow the [installation guide](/docs/installation) to get started.

    ### Database

    RubyAuth supports various databases. Set up your preferred database. In this guide, we’ll use PostgreSQL with the default database adapter.

    ```ruby title="config/initializers/better_auth.rb"
require "better_auth"

auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  base_url: ENV.fetch("BETTER_AUTH_URL", "http://localhost:3000"),
  database: :postgres,
  email_and_password: { enabled: true },
  social_providers: {
    google: {
      client_id: ENV.fetch("GOOGLE_CLIENT_ID"),
      client_secret: ENV.fetch("GOOGLE_CLIENT_SECRET")
    }
  }
)
```

    ### Email & Password

    Enable Email & Password authentication as shown below. Since WorkOS verifies each user’s email by default, this setup is similar to the default behavior. You can adjust it if needed. For more information, see [here](/docs/authentication/email-password).

    ```ruby
require "better_auth"

auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  base_url: ENV.fetch("BETTER_AUTH_URL", "http://localhost:3000"),
  database: :postgres,
  email_and_password: { enabled: true },
  social_providers: {
    google: {
      client_id: ENV.fetch("GOOGLE_CLIENT_ID"),
      client_secret: ENV.fetch("GOOGLE_CLIENT_SECRET")
    }
  }
)
```

    ### Social Providers (optional)

    Set up the social providers you used in WorkOS as follows. RubyAuth supports a wider range of providers, so you can add more if needed. Since WorkOS ensures emails are unique, configure `account.account_linking` in RubyAuth to ensure the same behavior.

    ```ruby

      client_id: ENV.fetch("GOOGLE_CLIENT_ID"),
      client_secret: ENV.fetch("GOOGLE_CLIENT_SECRET")
    }
  }
)
```

    Refer to the [section](#before-we-begin) mapping WorkOS features to RubyAuth. If a feature you used in WorkOS is available as a RubyAuth plugin, add it to the plugin options.   <Step>
    ## Generate Schema

    RubyAuth allows you to control your own database, and you can easily generate the appropriate schema for your auth instance using the CLI. For more information, see [here](/docs/concepts/cli).

    ### Default database adapter

    Run the `migrate` command to create the schema for your RubyAuth instance in the database.

    ```ruby title="Gemfile"
    bundle exec better-auth migrate --cwd . --config config/better_auth.rb --yes

```

    ### Other database adapters

    If you’re using a database adapter like Prisma or Drizzle, use the `generate` command to create the schema for your ORM. After that, run the migration with an external tool such as Drizzle Kit.

    ```ruby title="Gemfile"
    bundle exec better-auth generate --cwd . --dialect postgres --output db/better_auth/schema.sql --config config/better_auth.rb

```
  </Step>

  <Step>
    ## Migration Script

    ### Create Migration Script

    Create a migration script to import your user data from WorkOS into your database.

    ```ruby title="scripts/migration.rb"
# scripts/migrate_workos.rb
require "csv"
require "better_auth"

auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  base_url: ENV.fetch("BETTER_AUTH_URL"),
  database: :postgres,
  email_and_password: { enabled: true }
)

CSV.foreach("exports/workos_users.csv", headers: true) do |row|
  user = {
    email: row.fetch("email"),
    name: row["name"],
    email_verified: row["email_verified"] == "true"
  }

  # Prefer admin APIs when enabled; otherwise insert through your audited app import service.
  auth.api.create_user(user: user) if auth.api.respond_to?(:create_user)
end

# Existing workos sessions cannot be reused by RubyAuth. Plan a session invalidation window.
```

    <Callout type="info">
      **Notes**

      * When retrieving user data from WorkOS, you need to use their API, which is subject to rate limits. The example script includes a basic configuration, so adjust it as needed for your environment.
      * This migration script covers the common cases of managing users with email+password and social login. For features like SSO or CLI Auth, which are provided as plugins in RubyAuth, be sure to update the script based on the examples.
    </Callout>

    ### Run Migration Script

    ```bash title="Terminal"
    bundle exec ruby scripts/migration.rb
    ```

    🎉 Now that you’ve migrated your user data into your database, let’s look at how to update your application logic.
  </Step>

    ## Protecting Resources

    > Proxy (Middleware) is not intended for slow data fetching. While Proxy can be helpful for optimistic checks such as permission-based redirects, it should not be used as a full session management or authorization solution. - [Next.js docs](https://nextjs.org/docs/app/getting-started/proxy#use-cases)

    ### Middleware auth

    WorkOS provides Proxy (Middleware) authentication. RubyAuth doesn’t recommend protecting resources directly in middleware, so we don't provide dedicated helpers for that.

    ```ruby title="proxy.rb / app/controllers/application_controller.rb"
# Rails controller/helper example
before_action :require_current_user

def require_current_user
  @session = BetterAuth::Rails.current_session(request)
  redirect_to "/sign-in" unless @session
end
```

    In RubyAuth, for convenience rather than resource protection, the proxy (middleware) can be used as follows. This is supported in Next.js 15+ with the Node.js runtime.

    ```ruby title="proxy.rb"
# scripts/migrate_workos.rb
require "csv"
require "better_auth"

auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  base_url: ENV.fetch("BETTER_AUTH_URL"),
  database: :postgres,
  email_and_password: { enabled: true }
)

CSV.foreach("exports/workos_users.csv", headers: true) do |row|
  user = {
    email: row.fetch("email"),
    name: row["name"],
    email_verified: row["email_verified"] == "true"
  }

  # Prefer admin APIs when enabled; otherwise insert through your audited app import service.
  auth.api.create_user(user: user) if auth.api.respond_to?(:create_user)
end

# Existing workos sessions cannot be reused by RubyAuth. Plan a session invalidation window.
```

## Considerations

<strong className="underline italic">Password hashes</strong>

If you’ve been managing users with an email + password system, WorkOS does not provide an export of password hashes at this time. After migration, users will need to reset their passwords within your authentication system. Make sure to notify them of this change with sufficient lead time both before and after the migration.

<strong className="underline italic">Data syncing</strong>

WorkOS is a managed service and keeps your data in sync with your server through APIs or Webhooks. With RubyAuth, you fully own your authentication system and can manage data freely through the API. However, if you previously relied on Webhooks for synchronization, additional adjustments will be needed.

<strong className="underline italic">Downtime</strong>

WorkOS exposes data through its API, but with limitations such as the inability to export password hashes. Because of these constraints, performing a migration with zero downtime is challenging. Plan the migration carefully, allow enough buffer time, and communicate the expected impact to your users.

<strong className="underline italic">Active sessions</strong>

Existing active sessions will not be migrated. After the migration, users will need to sign in again, so be sure to notify them in advance.
