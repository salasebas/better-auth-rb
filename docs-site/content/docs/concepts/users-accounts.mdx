---
title: Users And Accounts
description: Learn how to update users, change email, manage passwords, delete users, list accounts, link accounts, unlink accounts, and handle OAuth tokens.
---

Beyond authenticating users, RubyAuth provides server-side APIs for managing user profiles, credentials, linked provider accounts, and account deletion.

The user table stores core authentication profile data. It can be extended with [additional fields](/docs/concepts/database#extending-core-schema) or by plugins. Accounts represent the user's authentication methods, such as credentials, Google, GitHub, or another OAuth provider, and store provider identifiers plus provider token metadata.

## Update User

### Update User Information

Use `update_user` to update profile fields on the current signed-in user. The endpoint accepts declared user fields such as `name`, `image`, and configured additional fields. Email cannot be changed through this route because email changes have their own verification flow.

```ruby
auth.api.update_user(
  headers: {"cookie" => cookie},
  body: {
    name: "Jane Doe",
    image: "https://example.com/avatar.png"
  }
)
```

Email cannot be changed with `update_user`; use `change_email`.

### Change Email

Email changes are disabled by default. Enable `user.change_email.enabled` and configure `email_verification.send_verification_email` so RubyAuth can verify the new address before updating the user.

```ruby
auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  user: {
    change_email: {
      enabled: true
    }
  },
  email_verification: {
    send_verification_email: ->(data, _request) { AuthMailer.verify_email(data).deliver_later }
  }
)
```

Request the change from the current session:

```ruby
auth.api.change_email(
  headers: {"cookie" => cookie},
  body: {
    newEmail: "new@example.com",
    callbackURL: "/settings"
  }
)
```

By default, RubyAuth sends a verification email to the new address. The user email is updated only after the verification link is used.

<Callout type="warning">
  Treat change-email confirmation and verification links as bearer credentials: anyone who possesses a valid link can use it. The links work across devices and do not require the session that requested the change. Redeeming the final verification link creates a session when one is not already present, even when `email_verification.auto_sign_in_after_verification` is `false`.
</Callout>

Change-email confirmation and verification links are single-use. RubyAuth records a hash of each issued token and consumes that record before performing the change. Deploying this behavior invalidates outstanding typed change-email links issued by an older version because those links do not have a corresponding verification record; users must request a new link. Ordinary email-verification links are unaffected.

#### Confirming With Current Email

For added security, require the current email to approve the change before RubyAuth sends verification to the new address. This optional confirmation helps protect accounts where the active session is present but the email change should still require control of the current inbox.

```ruby
auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  user: {
    change_email: {
      enabled: true,
      send_change_email_confirmation: lambda do |data, _request|
        AuthMailer.with(
          user: data.fetch(:user),
          new_email: data.fetch(:new_email),
          url: data.fetch(:url),
          token: data.fetch(:token)
        ).confirm_email_change.deliver_later
      end
    }
  },
  email_verification: {
    send_verification_email: ->(data, _request) { AuthMailer.verify_email(data).deliver_later }
  }
)
```

The flow is:

1. `change_email` sends a confirmation link to the current email.
2. The current email confirmation sends a verification link to the new email.
3. The new email verification updates the user email and marks it verified.

#### Updating Without Verification

Allow unverified users to update email immediately by enabling `update_email_without_verification`. This only applies when the current email is not verified.

```ruby
user: {
  change_email: {
    enabled: true,
    update_email_without_verification: true
  }
}
```

If a verification sender is configured, Better Auth sends a normal verification link for the new email after the update. When `update_email_without_verification` is false, the email is not updated until the new email is verified, even if the current email is unverified.

#### HTTP Usage

HTTP clients call:

```bash
curl -i http://localhost:3000/api/auth/change-email \
  -H 'content-type: application/json' \
  -H "cookie: $COOKIE" \
  -d '{"newEmail":"new@example.com","callbackURL":"/settings"}'
```

### Change Password

A user's password is stored on the credential account, not directly on the user row. Use `change_password` when the current user already has a credential password. Set `revokeOtherSessions` to invalidate other active sessions after the password changes.

```ruby
auth.api.change_password(
  headers: {"cookie" => cookie},
  body: {
    currentPassword: "old-password",
    newPassword: "new-password",
    revokeOtherSessions: true
  }
)
```

### Set Password

Use `set_password` for users who signed in with OAuth and do not have a credential password. This is server-only and requires a sensitive current session. If the user already has a password, RubyAuth rejects the request.

```ruby
auth.api.set_password(
  headers: {"cookie" => cookie},
  body: {
    newPassword: "password123"
  }
)
```

### Verify Password

Use `verify_password` when you need to confirm the user's current credential password before a sensitive server-side operation. OAuth-only users may not have a password; use email verification or fresh-session checks for those flows.

```ruby
auth.api.verify_password(
  headers: {"cookie" => cookie},
  body: {
    password: "password123"
  }
)
```

## Delete User

RubyAuth can hard-delete the current user and their sessions. The route is disabled by default; enable it explicitly before exposing account deletion.

```ruby
user: {
  delete_user: {
    enabled: true
  }
}
```

Delete with password when the user has a credential account:

```ruby
auth.api.delete_user(
  headers: {"cookie" => cookie},
  body: {
    password: "password123"
  }
)
```

### Adding Verification Before Deletion

For added security, configure `send_delete_account_verification` to send a verification token or generated callback URL before deleting the account. This is especially useful for OAuth-only users who cannot confirm deletion with a password.

```ruby
user: {
  delete_user: {
    enabled: true,
    send_delete_account_verification: lambda do |data, _request|
      AuthMailer.with(
        user: data.fetch(:user),
        token: data.fetch(:token)
      ).delete_account.deliver_later
    end
  }
}
```

Then complete deletion with the token:

```ruby
auth.api.delete_user(
  headers: {"cookie" => cookie},
  body: {
    token: params[:token]
  }
)
```

### Authentication Requirements

Deletion requires one of:

- The correct password.
- A valid delete-account verification token.
- A fresh session when no password or token is supplied.

When a password is provided, RubyAuth verifies the credential account before deleting. When a verification sender is configured, the delete route requires the token from that email before it deletes the user.

### Callbacks

Use callbacks for cleanup and audit side effects around deletion. `before_delete` runs before the user is removed; `after_delete` runs after deletion succeeds. A `before_delete` callback can abort deletion by raising an error or by using database hooks that return `false`.

```ruby
user: {
  delete_user: {
    enabled: true,
    before_delete: ->(user, _request) { AuditLog.create!(user_id: user["id"], action: "before_delete") },
    after_delete: ->(user, _request) { AuditLog.create!(user_id: user["id"], action: "after_delete") }
  }
}
```

## Accounts

Accounts represent credentials and linked OAuth providers. When a user signs in with a provider, RubyAuth creates or updates an account record containing provider IDs, scopes, token expiry metadata, and provider token values.

### List User Accounts

```ruby
accounts = auth.api.list_accounts(headers: {"cookie" => cookie})
```

The response omits sensitive token and password fields. OAuth accounts include provider identifiers, scopes, and expiry metadata that can be used to display connected accounts.

### Token Encryption

RubyAuth can encrypt stored OAuth access and refresh tokens before persisting them. Enable this when you store provider tokens and want database-at-rest protection controlled by your RubyAuth secret configuration.

```ruby
account: {
  encrypt_oauth_tokens: true
}
```

`get_access_token` returns decrypted access tokens when the current user has access.

```ruby
token = auth.api.get_access_token(
  headers: {"cookie" => cookie},
  body: {
    providerId: "github",
    accountId: "github-user-id"
  }
)

refreshed = auth.api.refresh_token(
  headers: {"cookie" => cookie},
  body: {
    providerId: "github",
    accountId: "github-user-id"
  }
)
```

### Account Linking

Account linking lets users associate multiple authentication methods with the same user. By default, linked accounts are accepted when the provider identity and email rules are satisfied. Configure account-linking rules when you need to trust specific providers, allow different emails, update user profile data after linking, or disable linking.

```ruby
auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  account: {
    account_linking: {
      enabled: true,
      trusted_providers: ["google", "github"],
      allow_different_emails: false,
      update_user_info_on_link: true
    }
  }
)
```

#### Forced Linking

Trusted providers can link accounts automatically when provider identity is verified and linking rules allow it. Use this carefully: trusting a provider that does not verify ownership can increase account takeover risk.

```ruby
account: {
  account_linking: {
    trusted_providers: ["google"]
  }
}
```

Forced linking is only appropriate for providers whose email or subject claims your app trusts. If the provider does not verify ownership, leave automatic linking disabled and require an explicit user action.

#### Manually Linking Accounts

Users already signed in can manually link a social account by starting a provider OAuth flow from their current session.

```ruby
link = auth.api.link_social(
  headers: {"cookie" => cookie},
  body: {
    provider: "github",
    callbackURL: "/settings/accounts"
  }
)
```

Native ID-token linking is available for providers with ID token verification. This is useful for mobile or native apps that already obtained a provider token through an SDK.

For redirect-based linking, send the user to the returned URL and let the provider callback complete the link. RubyAuth rejects unsafe callback URLs and accounts owned by another user.

### Account Unlinking

Use `unlink_account` to remove a provider account from the current user. Pass `accountId` when the user may have more than one account for the same provider.

```ruby
auth.api.unlink_account(
  headers: {"cookie" => cookie},
  body: {
    providerId: "github",
    accountId: "github-user-id"
  }
)
```

By default, Better Auth prevents unlinking the user's last account. Set `account.account_linking.allow_unlinking_all` only when your app has another recovery path.

Unlinking requires a fresh sensitive session and only removes the account owned by the current user.
