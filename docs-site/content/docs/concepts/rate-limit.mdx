Better Auth Ruby includes a built-in rate limiter to help manage traffic and slow down abusive clients before endpoint handlers run. In production, rate limiting is enabled by default unless you configure `rate_limit.enabled: false`; in development it is disabled unless explicitly enabled.

```ruby
auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  rate_limit: {
    enabled: true
  }
)
```

By default, general routes use `window: 10` and `max: 100`. Auth-sensitive paths such as `/sign-in`, `/sign-up`, `/change-password`, and `/change-email` use stricter upstream-compatible rules of 3 requests per 10 seconds. Other sensitive flows, such as password reset and verification email requests, use 3 requests per 60 seconds.

Plugin routes can also define their own rate-limit rules. For example, two-factor, magic-link, phone-number, and email-OTP flows apply plugin-specific limits where appropriate.

## Configuring Rate Limit

### Connecting IP Address

Rate limiting keys include the connecting client IP and route path. By default, Better Auth Ruby uses the IP resolved by your framework or Rack. It does not parse a raw `X-Forwarded-For` header itself, so the framework's trusted-proxy policy remains the source of truth.

If your ingress overwrites a single client-IP header, opt in to that header explicitly. Only do this for a header that clients cannot supply or preserve themselves.

```ruby
auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  advanced: {
    ip_address: {
      ip_address_headers: ["x-real-ip"]
    }
  },
  rate_limit: {
    enabled: true
  }
)
```

Comma-separated forwarded chains are rejected unless you also configure the proxy IP addresses or CIDR ranges that Better Auth may skip. Better Auth first verifies that the direct network peer is one of those trusted proxies, then walks the chain from right to left, skips trusted proxy hops, and selects the first untrusted address:

```ruby
advanced: {
  ip_address: {
    ip_address_headers: ["x-forwarded-for"],
    trusted_proxies: ["192.0.2.10", "10.0.0.0/24"]
  }
}
```

List the actual addresses of your proxies, not a broad private range that could also contain clients. Better Auth rejects the configured chain when the direct peer is missing or untrusted, or when any `trusted_proxies` entry is malformed. Still keep the application origin reachable only through those proxies, and configure every proxy to overwrite or sanitize forwarded IP headers. Explicit single-value headers remain an ingress trust decision and require the same sanitation even when `trusted_proxies` is not configured.

<Callout type="warning">
  This is a security-related compatibility change from earlier Better Auth Ruby releases, which trusted the leftmost `X-Forwarded-For` value by default. Deployments that relied on that behavior must either configure their framework's trusted proxies or opt in to an ingress-controlled header and, for chains, `trusted_proxies`.
</Callout>

When IP tracking is enabled but no trustworthy address can be resolved, requests share a fail-closed per-path bucket. This preserves rate-limit enforcement, but a proxy configuration error can cause unrelated clients to share a limit.

Disable IP tracking when another layer handles rate limiting. When IP tracking is disabled, Better Auth Ruby intentionally skips its built-in per-IP limiter.

```ruby
advanced: {
  ip_address: {
    disable_ip_tracking: true
  }
}
```

#### IPv6 Address Support

IPv6 addresses are normalized before becoming rate-limit keys. This prevents bypasses where the same IPv6 address is written in multiple valid forms, such as `2001:db8::1` and `2001:0db8:0000:0000:0000:0000:0000:0001`.

IPv4-mapped IPv6 addresses, such as `::ffff:192.0.2.1`, are normalized back to IPv4 so clients cannot switch representations to avoid limits.

#### IPv6 Subnet Rate Limiting

IPv6 clients often receive a whole address block, so limiting each full `/128` address can be too easy to evade. Configure IPv6 subnet grouping when you want requests from the same IPv6 allocation to share a rate-limit key:

```ruby
advanced: {
  ip_address: {
    ip_address_headers: ["x-real-ip"],
    ipv6_subnet: 64
  }
}
```

Common prefix lengths:

| Prefix | Use |
| --- | --- |
| `128` | Individual IPv6 address. |
| `64` | Typical home or business allocation. |
| `48` | Larger network allocation. |
| `32` | ISP-level allocation. |

IPv6 subnet grouping only affects IPv6 addresses. IPv4 addresses are always limited individually.

### Rate Limit Window

Configure a global `window` in seconds and `max` requests per window:

```ruby
rate_limit: {
  enabled: true,
  window: 60,
  max: 20
}
```

Use custom route rules for specific paths. Wildcards match route path prefixes and suffixes, and a rule value of `false` disables rate limiting for that path.

```ruby
rate_limit: {
  enabled: true,
  custom_rules: {
    "/magic-link/*" => {window: 60, max: 5},
    "/get-session" => false
  }
}
```

Custom rules can also be lambdas. The lambda receives the Rack request and the current rule so you can tighten, loosen, or disable the rule dynamically.

```ruby
rate_limit: {
  enabled: true,
  custom_rules: {
    "/admin/*" => lambda do |request, current_rule|
      current_rule.merge(max: 10)
    end
  }
}
```

### Storage

By default, rate-limit data is stored in memory. Each `BetterAuth.auth` instance owns an independent, thread-safe store capped at 100,000 live keys. Accepted requests use a rolling window; rejected requests do not extend it. Memory storage is not shared across auth instances, processes, machines, or serverless instances, so use database, secondary storage, or custom storage when counters must be shared.

**Using Database**

```ruby
auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  database: :memory,
  rate_limit: {
    enabled: true,
    storage: "database"
  }
)
```

Run migrations after enabling database storage so the `rateLimit` table exists. RubyAuth's generated physical table name is `rate_limits` by default, while the logical schema model remains `rateLimit`.

Database consumption is atomic and uses rolling windows. Expired-row cleanup is best effort and never fails a request. Cleanup accounts for every static rule; when a callable rule can return an unknown window, RubyAuth conservatively skips cleanup. `lastRequest` is stored as integer milliseconds for upstream schema compatibility, while Ruby configuration uses seconds.

**Using Secondary Storage**

If you configured a [secondary storage](/docs/concepts/database#secondary-storage), the rate limiter can store counters there. This is typically a good fit for Redis-like shared storage.

```ruby
auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  secondary_storage: redis_storage,
  rate_limit: {
    enabled: true,
    storage: "secondary-storage"
  }
)
```

When secondary storage implements atomic `increment(key, ttl)`, counters use a fixed window beginning with the first request. The TTL is applied only when the counter is created and is not extended by later requests; an over-limit response therefore reports the full configured window. Existing legacy JSON counters continue through their current rolling window and migrate in place to atomic counters after expiry. Without `increment`, enforcement remains best effort and logs one compatibility warning.

**Custom Storage**

Implement a custom storage object when database and secondary storage do not fit your deployment. For strict enforcement it must provide `consume(key, window:, max:)` and return `{allowed:, retry_after:}` atomically. `allowed` must be boolean, and `retry_after` must be `nil` or a finite nonnegative number. Legacy `get`/`set` implementations remain supported as best-effort compatibility and log one warning because concurrent requests can bypass a split read/write check.

```ruby
class RateLimitStorage
  def consume(key, window:, max:)
    # Atomically check and increment key in the backing store.
    {allowed: true, retry_after: nil}
  end
end

auth = BetterAuth.auth(
  secret: ENV.fetch("BETTER_AUTH_SECRET"),
  rate_limit: {
    enabled: true,
    custom_storage: RateLimitStorage.new
  }
)
```

## Handling Rate Limit Errors

For routed Rack requests, rate limiting runs after origin checks, plugin router middleware, and disabled-path checks, but before plugin `on_request`. A rejected request skips `on_request`; plugin `on_response` still wraps the `429`. Direct API calls do not pass through the Rack router and retain their existing caveat.

When a request exceeds the limit, Better Auth returns `429` and the `x-retry-after` header. The header value is the number of seconds until the client can retry.

```ruby
status, headers, body = auth.call(env)

if status == 429
  retry_after = headers.fetch("x-retry-after").to_i
  payload = JSON.parse(body.join)
end
```

Direct server-side API calls bypass the Rack request path unless you explicitly execute them as Rack responses. Client-initiated HTTP requests and API calls that execute through Rack observe the same response behavior.

### Schema

When `rate_limit.storage` is `"database"`, the schema includes the `rateLimit` model.

| Field | Type | Purpose |
| --- | --- | --- |
| `id` | string | Database ID. |
| `key` | string | Unique rate-limit key. |
| `count` | number | Number of requests in the current window. |
| `lastRequest` | number | Last request timestamp in epoch milliseconds. |

Customize the model name and fields with the `rate_limit` schema options before generating migrations.
