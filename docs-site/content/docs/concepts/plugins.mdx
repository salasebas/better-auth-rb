Plugins are a key part of RubyAuth, they let you extend the base functionalities. You can use them to add new authentication methods, features, or customize behaviors.

RubyAuth comes with many built-in plugins ready to use. Check the plugins section for details. You can also create your own plugins.

## Using a Plugin

Plugins can be a server-side plugin, a client-side plugin, or both.

To add a plugin on the server, include it in the `plugins` array in your auth configuration. The plugin will initialize with the provided options.

```ruby title="config/better_auth.rb"
require "better_auth"

auth = BetterAuth.auth(
  plugins: [
    BetterAuth::Plugins.two_factor,
    BetterAuth::Plugins.organization
  ]
)
```

RubyAuth lazy-loads plugin implementations through `BetterAuth::Plugins.method_missing`. Calling a factory such as `BetterAuth::Plugins.two_factor` loads the matching plugin file, then returns the plugin instance. This keeps the default boot path small while preserving a single `BetterAuth::Plugins.*` entrypoint.

<Callout type="info">
  RubyAuth does not ship an official browser client gem. Client-side plugin APIs
  from upstream Better Auth are JavaScript-only; call RubyAuth endpoints over
  HTTP from fetch, HTMX, Turbo, mobile clients, or server-rendered forms.
</Callout>

## Creating a Plugin

To get started, you'll need a server plugin.
Server plugins are the backbone of all plugins, and client plugins are there to provide an interface with frontend APIs to easily work with your server plugins.

<Callout type="info">
  If your server plugins have endpoints that need to be called from the client,
  expose HTTP endpoints and call them from your frontend or app client.
</Callout>

### What can a plugin do?

* Create custom `endpoint`s to perform any action you want.
* Extend database tables with custom `schemas`.
* Use a `middleware` to target a group of routes using its route matcher, and run only when those routes are called through a request.
* Use `hooks` to target a specific route or request. And if you want to run the hook even if the endpoint is called directly.
* Use `onRequest` or `onResponse` if you want to do something that affects all requests or responses.
* Create a custom `rate-limit` rule.

## Create a Server plugin

To create a server plugin, pass a `BetterAuth::Plugin` object.

The only required property is `id`, which is a unique identifier for the plugin.
Both server and client plugins can use the same `id`.

```ruby title="my_plugin.rb"
require "better_auth"

def my_plugin
  BetterAuth::Plugin.new(
    id: "my-plugin"
  )
end
```

<Callout>
  You don't have to make the plugin a function, but it's recommended to do so. This way, you can pass options to the plugin and it's consistent with the built-in plugins.
</Callout>

### Endpoints

To add endpoints to the server, you can pass `endpoints` which requires an object with the key being any `string` and the value being a `BetterAuth::Endpoint`.

To create an Auth Endpoint you'll need to instantiate `BetterAuth::Endpoint`.

RubyAuth wraps Rack-compatible request and response handling for endpoints. Endpoint blocks receive a `ctx` object that can read query, body, headers, params, and auth context.

```ruby title="my_plugin.rb"
require "better_auth"

def my_plugin
  BetterAuth::Plugin.new(
    id: "my-plugin",
    endpoints: {
      get_hello_world: BetterAuth::Endpoint.new(
        path: "/my-plugin/hello-world",
        method: "GET"
      ) do |ctx|
        ctx.json(message: "Hello World")
      end
    }
  )
end
```

Inside the `ctx` object, it'll provide another object called `context` that give you access better-auth specific contexts including `options`, `adapter`, `base_url` and more.

**Context Object**

* `app_name`: The name of the application. Defaults to "RubyAuth".
* `options`: The options passed to the RubyAuth instance.
* `tables`:  Core tables definition. It is an object which has the table name as the key and the schema definition as the value.
* `base_url`: the baseURL of the auth server. This includes the path. For example, if the server is running on `http://localhost:3000`, the baseURL will be `http://localhost:3000/api/auth` by default unless changed by the user.
* `session`: The session configuration. Includes `update_age` and `expires_in` values.
* `secret`: The secret key used for various purposes. This is defined by the user.
* `auth_cookie`: The default cookie configuration for core auth cookies.
* `logger`: The logger instance used by RubyAuth.
* `adapter`: This gives you ORM-like functions to interact with the database.
* `internal_adapter`: These are internal db calls that are used by RubyAuth. For example, you can use these calls to create a session instead of using `adapter` directly. `internal_adapter.create_session(user_id)`.
* `auth_cookies`: Cookie definitions for core auth cookies.
* `trusted_origins`: This is the list of trusted origins that you specified via `options.trusted_origins`.
* `trusted_origin?`: This is a helper function that allows you to quickly check whether a given url or path is trusted based on the trusted origins configuration.

**Rules for Endpoints**

* Makes sure you use kebab-case for the endpoint path
* Make sure to only use `POST` or `GET` methods for the endpoints.
* Any function that modifies a data should be a `POST` method.
* Any function that fetches data should be a `GET` method.
* Make sure to use the `BetterAuth::Endpoint` class to create API endpoints.
* Make sure your paths are unique to avoid conflicts with other plugins. If you're using a common path, add the plugin name as a prefix to the path. (`/my-plugin/hello-world` instead of `/hello-world`.)

### Schema

You can define a database schema for your plugin by passing a `schema` object. The schema object should have the table name as the key and the schema definition as the value.

```ruby title="my_plugin.rb"
require "better_auth"

def my_plugin
  BetterAuth::Plugin.new(
    id: "my-plugin",
    schema: {
      myTable: {
        fields: {
          name: {
            type: "string"
          }
        },
        model_name: "my_table" # optional if you want to use a different name than the key
      }
    }
  )
end
```

The migration generator, Rails and Hanami renderers, and MongoDB index setup
derive database additions from `Plugin#schema`. Use schema for plugin tables,
columns, and indexed fields; `Plugin#migrations` is not an upstream-backed
migration source in this Ruby port.

**Fields**

By default better-auth will create an `id` field for each table. You can add additional fields to the table by adding them to the `fields` object.

The key is the column name and the value is the column definition. The column definition can have the following properties:

`type`: The type of the field. It can be `string`, `number`, `boolean`, `date`.

`required`:  if the field should be required on a new record. (default: `true`)

`unique`: if the field should be unique. (default: `false`)

`references`: if the field is a reference to another table. (optional) It takes an object with the following properties:

* `model`: The table name to reference.
* `field`: The field name to reference.
* `on_delete`: The action to take when the referenced record is deleted. (default: `cascade`)

**Other Schema Properties**

`disable_migration`: prevents migration and schema generators from creating or
altering the table. The camel-case `disableMigration` spelling is also accepted
for compatibility with upstream plugin metadata. The table remains available to
runtime adapters, so use this only when another system manages its storage.
(default: `false`)

<Callout type="warn">
  Migration-disabled tables are not provisioned by RubyAuth. Operators must
  create the table, fields, foreign keys, and indexes themselves before the
  plugin uses them. If multiple logical schema entries share one physical table,
  setting either spelling to `true` on any contributor disables migrations for
  that entire physical table.
</Callout>

```ruby title="my_plugin.rb"
require "better_auth"

def my_plugin(opts)
  BetterAuth::Plugin.new(
    id: "my-plugin",
    schema: {
      rateLimit: {
        fields: {
          key: {
            type: "string"
          }
        },
        disable_migration: opts.dig(:storage, :provider) != "database"
      }
    }
  )
end
```

if you add additional fields to a `user` or `session` table, the API responses that return that model include the additional fields.

```ruby title="my_plugin.rb"
require "better_auth"

def my_plugin
  BetterAuth::Plugin.new(
    id: "my-plugin",
    schema: {
      user: {
        fields: {
          age: {
            type: "number"
          }
        }
      }
    }
  )
end
```

This will add an `age` field to the `user` table and all `user` returning endpoints will include the `age` field.

<Callout type="warn">
  Don't store sensitive information in the `user` or `session` table. Create a new table if you need to store sensitive information.
</Callout>

### Hooks

Hooks are used to run code before or after an action is performed, either from a client or directly on the server. You can add hooks to the server by passing a `hooks` object, which should contain `before` and `after` properties.

```ruby title="my_plugin.rb"
require "better_auth"

def my_plugin
  BetterAuth::Plugin.new(
    id: "my-plugin",
    hooks: {
      before: [
        {
          matcher: ->(context) { context.headers["x-my-header"] == "my-value" },
          handler: ->(ctx) {
            ctx.context.plugin_state = {source: "my-plugin"} if ctx.context.respond_to?(:plugin_state=)
            {context: ctx}
          }
        }
      ],
      after: [
        {
          matcher: ->(context) { context.path == "/sign-up/email" },
          handler: ->(ctx) {
            ctx.json(message: "Hello World")
          }
        }
      ]
    }
  )
end
```

### Middleware

You can add middleware to the server by passing a `middlewares` array. This array should contain middleware objects, each with a `path` and a `middleware` property. Unlike hooks, middleware only runs on `api` requests from a client. If the endpoint is invoked directly, the middleware will not run.

The `path` can be either a string or a path matcher.

If you raise a `BetterAuth::APIError` from the middleware or return a Rack response tuple, the request will be stopped, and the response will be sent to the client.

```ruby title="my_plugin.rb"
require "better_auth"

def my_plugin
  BetterAuth::Plugin.new(
    id: "my-plugin",
    middlewares: [
      {
        path: "/my-plugin/hello-world",
        middleware: ->(ctx) {
          ctx.headers["x-plugin-middleware"] = "yes"
          nil
        }
      }
    ]
  )
end
```

### On Request & On Response

Additional to middlewares, you can also hook into right before a request is made and right after a response is returned. This is mostly useful if you want to do something that affects all requests or responses.

#### On Request

The `onRequest` function is called right before the request is made. It takes two parameters: the `request` and the `context` object.

Here's how it works:

* **Continue as Normal**: If you don't return anything, the request will proceed as usual.
* **Interrupt the Request**: To stop the request and send a response, return a Rack response tuple.
* **Modify the Request**: You can also return a modified `request` object to change the request before it's sent.

```ruby title="my_plugin.rb"
require "better_auth"

def my_plugin
  BetterAuth::Plugin.new(
    id: "my-plugin",
    on_request: lambda do |request, _context|
      request.env["HTTP_X_PLUGIN_REQUEST"] = "yes"
      {request: request}
    end
  )
end
```

#### On Response

The `onResponse` function is executed immediately after a response is returned. It takes two parameters: the `response` and the `context` object.

Here's how to use it:

* **Modify the Response**: You can return a modified response object to change the response before it is sent to the client.
* **Continue Normally**: If you don't return anything, the response will be sent as is.

```ruby title="my_plugin.rb"
require "better_auth"

def my_plugin
  BetterAuth::Plugin.new(
    id: "my-plugin",
    on_response: lambda do |response, _context|
      response[1]["x-plugin-response"] = "yes"
      {response: response}
    end
  )
end
```

### Rate Limit

You can define custom rate limit rules for your plugin by passing a `rateLimit` array. The rate limit array should contain an array of rate limit objects.

```ruby title="my_plugin.rb"
require "better_auth"

def my_plugin
  BetterAuth::Plugin.new(
    id: "my-plugin",
    rate_limit: [
      {
        path_matcher: ->(path) { path == "/my-plugin/hello-world" },
        max: 10,
        window: 60
      }
    ]
  )
end
```

### Trusted origins

If you're building custom plugins or endpoints, you can use the `trusted_origin?` method available on the auth context to validate URLs against your trusted origins configuration.
This ensures your custom endpoints respect the same security settings as RubyAuth's built-in endpoints.

```ruby title="my_plugin.rb"
require "better_auth"

def my_plugin
  BetterAuth::Plugin.new(
    id: "my-plugin",
    trusted_origins: [
      "http://trusted.com"
    ],
    endpoints: {
      get_trusted_hello_world: BetterAuth::Endpoint.new(
        path: "/my-plugin/hello-world",
        method: "GET"
      ) do |ctx|
        unless ctx.context.trusted_origin?(ctx.query["url"], allow_relative_paths: false)
          raise BetterAuth::APIError.new("FORBIDDEN", message: "origin is not trusted.")
        end

        ctx.json(message: "Hello World")
      end
    }
  )
end
```

See the [trusted origins and security](/docs/reference/security#trusted-origins) docs for more info.

### Server-plugin helper functions

Some additional helper functions for creating server plugins.

#### `getSessionFromCtx`

Allows you to get the client's session data by passing the auth middleware's `context`.

<UnderDevelopment>
  RubyAuth does not currently expose a direct `getSessionFromCtx` helper.
  Server plugins can call `BetterAuth::Session.find_current(ctx)` or use
  `ctx.context.internal_adapter` when they need lower-level session data.
</UnderDevelopment>

#### `sessionMiddleware`

A middleware that checks if the client has a valid session. If the client has a valid session, it'll add the session data to the context object.

<UnderDevelopment>
  RubyAuth does not currently expose a standalone `sessionMiddleware` helper.
  Plugin middleware can inspect cookies and call `BetterAuth::Session.find_current`
  or delegate to an endpoint that already requires a session.
</UnderDevelopment>

#### `requireResourceOwnership`

A middleware that loads a resource by ID and verifies that it belongs to the authenticated user. Use it after `sessionMiddleware` for endpoints that operate on user-owned models.

<UnderDevelopment>
  RubyAuth does not currently expose a packaged `requireResourceOwnership`
  helper. Implement the ownership check in your endpoint or middleware using
  `ctx.context.adapter` or `ctx.context.internal_adapter`.
</UnderDevelopment>

Use `ownerField` if your model stores the owner under a different field name, and `notFoundError` or `forbiddenError` if you want to return plugin-specific error codes.

#### `requireOrgRole`

A middleware that verifies the authenticated user is a member of a specific organization and, optionally, has one of a set of allowed roles. Use it after `sessionMiddleware` for organization-scoped endpoints.

<UnderDevelopment>
  RubyAuth does not currently expose a packaged `requireOrgRole` helper. The
  organization plugin has server-side membership and permission APIs; custom
  plugin endpoints should call those APIs or perform the membership query
  directly.
</UnderDevelopment>

If `allowedRoles` is omitted, any organization member is accepted. Members with multiple roles are authorized when any one of their roles matches.
