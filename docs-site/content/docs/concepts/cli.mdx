Better Auth Ruby uses Rails generators, the `better-auth` CLI, and Ruby tasks.

## Init

Install Better Auth into an existing Ruby app with the unified CLI (non-interactive):

```bash
better-auth init --cwd . --framework rails
better-auth init --cwd . --framework hanami|sinatra|roda|rack
better-auth init --cwd . --detect-framework
```

- `--cwd` is required.
- Pass exactly one of `--framework` or `--detect-framework`.
- `rack` is never auto-detected; use `--framework rack` for generic Rack apps.
- Rails/Hanami/Sinatra/Roda installs delegate to each framework package's generator or rake task.
- Rack installs write a minimal `config/better_auth.rb` and migration directory.

The Rails install generator also remains available directly:

```bash
bin/rails generate better_auth:install
```

```ruby title="config/routes.rb"
mount BetterAuth::Rails.app => "/api/auth"
```

### Options

Use `--force` to overwrite CLI-owned Rack scaffold files. Framework-specific
options (`--secret`, `--base-url`, `--database-dialect`) are validated when
present but not auto-filled.

## Explicit flags

Config-backed commands require `--cwd`. Pass `--config PATH` or add
`--discover-config` to search conventional paths under `--cwd`:

1. `config/better_auth.rb`
2. `config/auth.rb`
3. `better_auth.rb`
4. `auth.rb`

There are no silent defaults for working directory, config path, or SQL dialect.

## Generate

Generate Rails installation files:

```bash
bin/rails generate better_auth:install
```

Generate database migrations:

```bash
bin/rails generate better_auth:migration
```

The migration generator reads your Better Auth configuration and includes core tables, plugin tables, custom table names, custom fields, and optional rate limit schema.

For non-Rails Rack apps, use the `better-auth` CLI:

```bash
better-auth generate --cwd . --config config/better_auth.rb --dialect postgres --output db/better_auth/schema.sql
```

Or discover config under the project root:

```bash
better-auth generate --cwd . --discover-config --dialect postgres --output db/better_auth/schema.sql
```

Relative `--config` and `--output` paths resolve against `--cwd`.

### Options

Rails generators accept standard Rails generator options, such as `--skip` options provided by Rails itself. Better Auth-specific schema is read from the configured initializer.

For the standalone CLI, `--cwd`, (`--config` or `--discover-config`), `--dialect`, and `--output` are all required on `generate`.

## Migrate

Run migrations with Rails:

```bash
bin/rails db:migrate
```

For a non-Rails Rack app, use the CLI against a Ruby config file:

```bash
better-auth migrate status --cwd . --config config/better_auth.rb
better-auth migrate --cwd . --config config/better_auth.rb --yes
```

You can also generate SQL programmatically and apply it with your own migration tool.

```ruby
sql = BetterAuth::Schema::SQL.generate(auth.options, adapter: :postgres)
File.write("db/auth_schema.sql", sql)
```

### Options

Use your database adapter or migration tool options when applying generated SQL outside Rails.

For the standalone CLI, use `--cwd`, (`--config` or `--discover-config`), and `--yes` on `migrate`.

## Info

Inspect the active configuration from Ruby:

```ruby
auth.options.base_path
auth.options.session
auth.api.endpoints.keys
BetterAuth::Schema.auth_tables(auth.options).keys
```

Run `better-auth doctor` against a project config:

```bash
better-auth doctor --cwd . --config config/better_auth.rb
better-auth doctor --cwd . --discover-config
better-auth doctor --cwd . --config config/better_auth.rb --json
```

For machine-readable diagnostics, use `better-auth info --json`:

```bash
better-auth info --cwd . --json
better-auth info --cwd . --config config/better_auth.rb --json
```

`info --json` returns Ruby and gem versions, resolved auth table names,
endpoint counts, and doctor findings without serializing sensitive config
values.

For route-level inspection, Rails can list the mounted auth routes:

```bash
bin/rails routes | grep auth
```

### Output

The useful inspection output is the resolved Ruby configuration, registered endpoint names, and generated auth tables.

### Options

Use the same Ruby object inspection patterns in console, tests, or diagnostics.

### Examples

Basic usage:

```ruby
auth.api.endpoints.keys
```

Custom config path:

```bash
better-auth doctor --cwd . --config config/better_auth.rb
```

Discover config from the project root:

```bash
better-auth doctor --cwd . --discover-config
```

JSON output:

```bash
better-auth info --cwd . --config config/better_auth.rb --json
```

## Secret

Generate a production-ready secret with the CLI:

```bash
better-auth secret
```

The command prints `BETTER_AUTH_SECRET=<64 lowercase hex chars>` for easy copy
into `.env` or your framework credentials.

Without the CLI gem, you can still generate a secret with:

```bash
openssl rand -base64 32
```

Store the value in your framework's credential or environment system:

```ruby
secret: ENV.fetch("BETTER_AUTH_SECRET")
```

## Common Issues

If generated migrations miss plugin tables, make sure the initializer includes the plugin before running the migration generator.

If callbacks generate untrusted callback URL errors, configure `base_url` and `trusted_origins` to match the public origin used by your app.

If requests do not keep sessions, forward `Set-Cookie` from auth responses and include the cookie header on later server-side calls.

If the CLI reports missing flags, pass `--cwd` and either `--config` or `--discover-config` on config-backed commands.

## Command reference

| Command | Required flags |
| --- | --- |
| `init` | `--cwd`, (`--framework` or `--detect-framework`) |
| `upgrade` | `--cwd` |
| `generate` | `--cwd`, (`--config` or `--discover-config`), `--dialect`, `--output` |
| `migrate` | `--cwd`, (`--config` or `--discover-config`), `--yes` |
| `doctor` / `mongo indexes` | `--cwd`, (`--config` or `--discover-config`) |
| `info` | `--cwd` |
| `secret` | none |
